# spec-093 fix report (Lane-W agent)

## GROUP A ŌĆö De-fixture Query Correctness (engine_bridge.py)

### Finding #1 ŌĆö get_educate_targets queries non-existent node type

**Resolution**: FIXED. Community nodes live in the XGI hypergraph (`CommunityHypergraph`),
NOT as main-graph nodes. `WorldState.to_graph()` creates only: social_class, territory,
organization, key_figure, institution, industry. The `_communities_in_territory` helper
queried `_node_type == "community"` which never exists in production graphs.

**Correct target type**: `social_class` nodes in the territory (via `_nodes_in_territory`
filtered to `_node_type == "social_class"`). The educate verb raises `education_pressure`
on community hyperedges, but since the bridge doesn't access the XGI hypergraph, targets
are presented as social_class nodes with their real `agitation` data. Consciousness fields
(r, l, f) are noted as pending hypergraph integration.

### Finding #2 ŌĆö Extractive-edge filter reads wrong attribute

**Resolution**: FIXED. The engine writes `edge_mode` as a **separate** graph edge attribute
from `edge_type`. `EdgeTransitionSystem` calls `graph.update_edge(..., edge_mode=new_mode)`
where `new_mode` is an `EdgeMode` value (extractive/antagonistic/etc.). The bridge code was
reading `edge_type` (which carries `EdgeType` values like "exploitation") and comparing
against `EdgeMode` values ("extractive") ŌĆö always empty.

**Correct attribute**: `edge_mode` (checked first, with `edge_type`/`_edge_type` as fallback).

### Finding #3 ŌĆö extraction_intensity read from wrong node type

**Resolution**: FIXED. `ProductionSystem._update_extraction_intensities` writes
`extraction_intensity` ONLY onto Territory nodes (line 254:
`graph.update_node(node.id, extraction_intensity=intensity)` where node is a territory).
The old code read it from social_class/organization nodes returned by `_nodes_in_territory`,
which excludes territory nodes.

**Correct source**: Territory node attributes (`graph.nodes[territory_id]`).

### Finding #4 ŌĆö get_mobilize_targets uses wrong enum string

**Resolution**: FIXED. `OrgType.CIVIL_SOCIETY.value` is `"civil_society"` (no `_org` suffix).
The old code filtered `"civil_society_org"` which matched nothing.

**Correct value**: `"civil_society"` (matching `OrgType.CIVIL_SOCIETY`).

## CROSS-LANE ŌĆö Finding #6 (E-lane, NOT fixed by W-lane)

### Finding #6 ŌĆö from_graph() crashes on faction/sovereign/community nodes

**Assessment**: **BLOCKING cross-lane dependency ŌĆö bridge triggers a latent production crash.**

`hydrate_state()` (engine_bridge.py:388) calls `WorldState.from_graph(graph, tick=...)`.
`from_graph()` (world_state.py:484-505) dispatches by `_node_type` but has NO branch for
`faction`, `sovereign`, `community`, `balkanization_faction`, or `hex` nodes. These fall
through to `SocialClass(**entity_data)` with `extra="forbid"` and CRASH.

**Reproduction**: Any game with spec-070 balkanization data (faction/sovereign nodes in the
graph) will crash on every `hydrate_state()` call ŌĆö which is invoked by every bridge method
that reads game state (get_economy, get_educate_targets, get_snapshot, etc.).

**Why tests don't catch it**: `_patched_hydrate_state` in the test file patches
`hydrate_state` to return a mock WorldState, bypassing `from_graph` entirely.

**E-lane fix needed**: Add branches in `WorldState.from_graph()` for `_node_type` in
{faction, sovereign, balkanization_faction, hex, community} ŌĆö either reconstructing them
into appropriate model types or skipping them gracefully.
