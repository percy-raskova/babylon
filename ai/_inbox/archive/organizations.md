Good instinct — let me trace the chain, because the smell isn't the shim itself, it's what the shim's *test situation* tells you about where the organizational semantics actually live now.

**What we found, restated as evidence.** `OrganizationComponent` is two fields — `cohesion` and `cadre_level`, both `Probability`-constrained, frozen, validated. Feature 031 deprecated it in favor of `Organization` and its subtypes (StateApparatus, Business, PoliticalFaction, CivilSocietyOrg). When I swept for consumers today: **zero production callers**. Its only consumer is its own 270-line contract test suite — creation, validation, boundary values, immutability, serialization round-trip. Meanwhile the deprecation warning had been firing on *every single `import babylon.models`* since Feature 031, and nobody noticed until we flipped first-party DeprecationWarnings to errors this afternoon.

Each of those facts is a distinct smell:

**1. A deprecation that never completes means the migration was never verified complete.** Deprecation shims are supposed to be scaffolding with a demolition date. This one fossilized. The usual reason a shim outlives its migration is that nobody ever proved the successor covers the shim's semantics — so nobody felt safe deleting it. The question "can we delete OrganizationComponent?" is really the question "does Organization fully model organizational capacity?" — and the fact that the shim still stands suggests nobody has been able to answer yes.

**2. The best behavioral contract in the org domain protects a corpse.** Your test estate is a map of what the system guarantees. Right now, the most rigorous, thorough spec of organizational-capacity semantics — typed bounds, immutability, serialization — is attached to the *dead* class. The *live* org state, meanwhile, is spread across places with much weaker contracts: graph node attribute dicts (which systems read directly — `graph.nodes[id]["cohesion"]`, no Pydantic enforcement at mutation time), `org_snapshot` columns (`coherence`, `reputation`, `opsec`, `cadre_count`, `sympathizer_labor` — schema-typed but semantically unpinned), and `OODAProfile` stashed on graph nodes. The concepts survived — `cohesion` became a column, `cadre_level` became `cadre_count`/`cadre_labor` — but I can't point you to an equivalent entity-level contract test proving the *live* representation enforces the bounds the dead one did. Apply your own rewrite test: "what would prove a rewrite of org capacity correct?" Today the honest answer is the shim's tests. That's exactly backwards.

**3. The suppress-and-import dance is an IOU.** To keep testing the shim under the new warnings-as-errors regime, we had to write `with warnings.catch_warnings(): ignore` around the import. Any time a test must explicitly ignore a deprecation to reach its subject, the test is propping open a door the code declared closed. It's legitimate scaffolding — but it's dated scaffolding, and the date was Feature 031.

**4. The org domain's *other* test coverage was vacuous in the same sweep.** Not a coincidence: the property harness's `worldstate_strategy` never generated organization nodes at all, so the consequence-after-actions invariant quantified over an empty set for months (fixed today). Add the pattern from the Phase G list — org-adjacent code that's tested-but-unwired — and `Organization.is_institution` carrying its *own* deprecation pointing at Institution. So Organization has now been the target of two half-finished migrations (component→entity, org⟷institution split), each leaving a stub. The entity is a landing zone that keeps accreting arrivals without consolidating them.

**Why this converges on "build out Organization":** the class is about to take on *more* load, not less — DoctrineSystem accumulates per-org doctrine state (ADR073, with bifurcation feedback coming in DT Unit 6), P19 gives orgs pole readings, OODA dispatches per-org verbs. Piling that onto an entity whose foundational capacity semantics live as unconstrained graph-dict floats is how you get the next generation of silent drift.

Concretely, "build out" would mean: pull the capacity semantics (`cohesion`, cadre, `opsec`, `coherence`, `reputation`) into the Organization entity as constrained types with the same rigor the shim had; make sure `to_graph`/`from_graph` round-trips them (the known gotcha — `from_graph` silently drops non-core attrs — is exactly where this would rot); **port the shim's contract tests to the entity** (they're genuinely good — the contract should outlive the implementation); *then* delete the shim and its suppress-dance test. The deletion itself is production surgery, so it's owner-gated Phase G territory — but the test refactor is what surfaced the case for it.

One caveat: I'm working from today's sweep and the snapshot schema, not a fresh read of the entity model — before acting on this, the first step is verifying exactly which capacity fields `Organization` already carries with real constraints versus which exist only as graph attrs and columns. But the shape of the smell is solid: when a refactor finds the strongest tests guarding the deadest code, the live code is underspecified.
