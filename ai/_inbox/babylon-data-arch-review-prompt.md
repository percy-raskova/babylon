Paste everything below the line into Claude Code from the repo root. Read-only review; the only writes are report files.

You are running a read-only architecture, correctness, and operations review of Babylon's entire data architecture: the Postgres runtime, the Django bridge in `web/`, and the reference-data pipeline that feeds them. You will produce reports under `reports/data-arch-review/` and change nothing else — no code edits, no DDL, no migrations, no database writes.

## Phase 0 — Recon (you, the orchestrator, before spawning anything)

1. Map the Postgres surface. Find every module that imports psycopg or otherwise opens connections / executes SQL (start at `src/babylon/persistence/`, but do not assume it is exhaustive — check the intelligence/pgvector layer, `archival.py` and its DuckDB consumers, and any tools/ scripts). Find the engine migration files (the `00*.sql` set and whatever applies them), the compose files and `postgresql.conf` tuning, and the test fixtures that stand up the :5433 database.
2. Map the Django surface. In `web/`: the settings module(s) and `DATABASES` config, `INSTALLED_APPS`, every `models.py` with each model's `Meta.managed` and `db_table`, the Django migrations directory, urls/views/serializers (note whether DRF or plain views), middleware (CORS/CSRF), and how the web layer launches or monitors simulation runs.
3. Map the reference-data surface. The read-only SQLite reference DB (`data/sqlite/marxist-data-3NF.sqlite`) and every site that opens it; the ingest pipelines (Census/QCEW/BEA, wherever they live); `defines.yaml` and its consumers; the Parquet archive layout and the DuckDB read path.
4. Read the binding documents in-repo before partitioning: ADR031 (tick-keyed temporal tables), ADR033 (deterministic simulation), ADR036 (orphan schema audit), ADR045 (QCEW normalization), ADR046 (BEA IO ingest), ADR047 (per-tick read cache), ADR050 (suppression imputation), ADR053 (storage scaling program), ADR027 (frontend stack), plus the spec-037 web-schema material if present, and Constitution clauses II.6 (no DB I/O during tick), II.11 (subsystem table ownership), III.7 (hash chain), III.11 (Loud Failure), Amendment L (byte-identity), and the Program 14 layering rules.
5. Build the table-ownership map: for every table in the database, record its owner — engine (`00*.sql`), Django (Django migrations), or contested. This map is the keystone input for Lanes 1, 5, and 7. Write the full inventory (files found, lane assignments, ownership map) to `reports/data-arch-review/00-recon.md`.
6. Confirm the lane partition. The eight lanes below are mandatory and may not be merged. Add a lane only if recon surfaced data-architecture surface none of them covers.

## Phase 1 — Parallel review lanes

Launch ALL eight lane subagents in parallel: a single message containing one Task invocation per lane. Subagents share no context with you or with each other, so every brief must be fully self-contained — include the relevant file paths from recon, the ownership map, the lane scope verbatim, the finding format, and the output path. Each subagent is read-only and writes its report to `reports/data-arch-review/NN-<lane>.md`.

**Lane 1 — Schema and migrations (engine side).** Migration-rerun idempotency: every `00*.sql` re-executes at runner start, so hunt for non-idempotent DDL and for view definitions living anywhere other than the canonical always-run file (ADR053 ruled views must, because they bind to table OIDs across renames). Partitioning correctness: LIST by session_id, DEFAULT partitions catching hookless writers, per-session partitions created at initialize_session. The (tick, entity_id) identity model of ADR031 as amended by ADR053's delta-plus-checkpoint persistence. DDL vs frozen Pydantic models, field by field — and whether constrained types (Probability, Currency, Intensity) exist as DB-level CHECKs or only in Python. Orphan tables and columns (rerun the ADR036 audit). `tick_commit` as dense spine and queryable hash chain.

**Lane 2 — Determinism and the hash chain (highest stakes).** Anything that can break byte-identity or the III.7 hash: SELECTs feeding simulation or aggregation logic without a total ORDER BY; float aggregation performed in SQL where row order affects the result — note that ADR053's byte-identity argument holds only while hex values are uniform within a county, so flag every site where that assumption is load-bearing; now()/CURRENT_TIMESTAMP/random()/sequence values leaking into persisted or hashed state; the as-of fill-forward views (v_hex_state_asof) resolving ties nondeterministically; synchronous_commit=off scoped to anything beyond the test role; parallel-query or planner effects on result order. Additionally, audit every write path that originates outside the engine — Django, operator tools, ingest scripts — and confirm none can land in a table that feeds the hash. Every finding in this lane must include a concrete failure scenario, not a suspicion.

**Lane 3 — Transactions, sessions, and the tick boundary.** Is one tick one atomic unit, and what state exists after a mid-tick crash — is `tick_commit` written last, and are partial writes reachable by readers? get_last_committed_tick semantics for pre- vs post-0029 sessions. Isolation level: set explicitly or assumed. Connection lifecycle and pooling for the engine vs the Django bridge (:5433 tests / :5432 web), including Django's transaction posture (ATOMIC_REQUESTS or explicit) wherever it touches shared tables — ownership questions belong to Lane 7; here, atomicity and isolation only. Error paths: ADR019 exception hierarchy vs bare excepts; DiskFull-class failures surfacing loudly per III.11 rather than being swallowed.

**Lane 4 — Performance and the per-tick I/O rule.** II.6 forbids DB I/O during the tick: verify the ReferenceDataCache guard (the test that patches the connection constructor after hydrate) still holds, and hunt for reference-data reads added since spec-069 that bypass hydrate and silently reintroduce per-tick I/O. Index diet vs the actual hot queries — enumerate the per-tick query set first, then check indexes against it, not the reverse. Write batching and delta-envelope efficiency. qa:storage-budget coverage of new tables. pgvector index type and parameters, and whether any Archive query can ever sit on the tick path. The DEFAULT partition as a silent performance sink for hookless writers.

**Lane 5 — Layering and boundaries.** Program 14 import-linter reality check: persistence imported only from layers permitted to see it. No engine math or domain logic embedded in SQL views or queries — every formula must trace to code, not DDL. II.11 table ownership: one declared read path per table family, no parallel SQL reaching the same tables from bridges or tools. The intelligence layer writes nothing into Ledger tables, and nothing AI-generated flows into hashed state (III.11).

**Lane 6 — Tests, ops, and config.** :5433 isolation and fixture hygiene under parallel test execution — schema setup per run, leakage between tests, and whether Django's test setup shares or collides with the engine fixtures on :5433. Migrations exercised in the correct CI lane (dev fast-fail vs main heavy, Program 15). Whether Playwright e2e is the only coverage of the web write paths. Compose pin (postgis:16-3.4 + pgvector, shared by web per ADR053) and postgresql.conf drift against what the determinism lane assumes. BABYLON_PG_DATA, secrets, and connection-string handling across engine and Django. The archival verify-before-DROP-PARTITION path: what happens if verification passes falsely. Backup and restore story for the local archives directory. Anything running as superuser that should not.

**Lane 7 — Django bridge (`web/`).** Schema ownership first, because two migration systems share one database: enumerate every table Django's migration files create or alter, diff that set against what the engine's `00*.sql` owns, and treat any overlap as P0 — an idempotent SQL rerun and Django's `migrate` state machine making conflicting claims on the same table is the failure mode that corrupts silently. Any Django model over an engine-owned table must be `managed = False` (or absent, with access through a declared read path); confirm a fresh `migrate` on an existing database cannot mutate engine schema. Model drift: Django model fields vs live DDL vs the engine's Pydantic models for every shared table, including precision through serialization (Currency/Probability surviving JSON without float coercion). Read-path correctness: post-0029, hex-history reads MUST go through `v_hex_state_asof` — raw `dynamic_hex_state` is sparse — so audit every Django query touching hex state; and the API must expose only ticks ≤ get_last_committed_tick, never partial in-flight writes. ORM hygiene: N+1s missing select_related/prefetch_related, unbounded querysets against the partitioned per-tick families, pagination, explicit ordering wherever the cockpit depends on order. Write boundary: enumerate every write Django performs; each must target Django-owned tables only, nothing Django writes may reach hash-feeding state (III.7, III.11), and ad-hoc raw SQL into other subsystems' tables violates II.11. Config and serving: DATABASES pinned to :5432 for web and never :5433, CONN_MAX_AGE and pooling posture, DEBUG / SECRET_KEY / ALLOWED_HOSTS sourcing, CORS scoped to the Vite origin (:5173), CSRF posture on the cockpit API. Run orchestration: how the web layer launches simulation runs — synchronous-request risk, timeouts, zombie runs, and whether two concurrent launches can write the same session.

**Lane 8 — Reference data and ingest.** The read-only SQLite reference DB: verify it is actually opened read-only (URI mode=ro or equivalent) at every open site, and that every open site is a declared II.11 path — the county_aggregation fetchers and the ReferenceDataCache hydrate, nothing ad-hoc. Ingest pipelines (QCEW normalization per ADR045, suppression imputation per ADR050, BEA IO per ADR046): idempotent and deterministic — same raw inputs produce byte-identical outputs — with provenance recorded; flag any manual, undocumented step between raw source and the reference DB. Coefficient discipline: every tunable constant belongs in `defines.yaml`; grep the ingest code, SQL, and bridge for hardcoded coefficients (the QCEW×0.33 population fallback is a known literal — find where it lives and whether it is define-sourced). The archive read path: Parquet+zstd layout, whether verify-before-purge would actually catch a truncated or corrupt export, and whether DuckDB fill-forward reconstruction is documented and tested against a live-database oracle. Finish with a data-flow map — SQLite → cache → bridge → Postgres → Parquet → DuckDB — flagging any hop that bypasses a declared interface.

**Finding format, mandatory for every lane:** `[P0|P1|P2|P3] path/to/file.py:LINE — one-line claim — evidence: the actual code or DDL, quoted — why it matters, citing the ADR or Constitution clause — recommended fix.` No finding without cited code: if you cannot point at a line, record it as an open question, not a finding (Loud Failure). A lane with zero findings must say so explicitly and enumerate what was checked — an empty report is a failed report.
