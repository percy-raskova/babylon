# Emacs vs Neovim vs Python TUI — and OCaml vs Rust for the Algebra Layer

*An honest research synthesis for the Babylon terminal Archive client and the Lawverian algebra-layer question.*

**Date:** 2026-07-21 · Prepared for BD review.
**Research method:** five parallel research streams — **S1** OCaml-vs-Rust, **S2** Neovim, **S3** Emacs, **S4** the in-repo Babylon seam / tests / ADRs, **S5** Textual + terminal-graphics — followed by an **adversarial critique pass** over the draft that re-checked every load-bearing claim, and a **Windows-constraint amendment pass (Amendment AA, ADR108)** verifying each candidate's Windows posture against primary sources (Part III; tagged **W1–W7**). Factual claims are tagged to a stream (S1–S5), to the Windows pass (W1–W7), or to a repo file by path; judgment calls are marked **[judgment]**.
**No code was modified** in producing this report. The only artifact written is this document. All in-repo figures (LOC, line numbers, file counts) were re-counted on the working branch at the time of writing (S4).

## Executive Summary

**Recommendation, in one breath:** Ship the existing **Textual** client for v1.0. Treat **Neovim** as the only credible *second* client and build it — if at all — *after* 1.0 as a Python-owns-state / nvim-as-renderer hybrid, justified mainly because you are personally Neovim-native. Do **not** adopt **Emacs** for this project's stated audience. And do **not** introduce **OCaml**: keep the Lawverian algebra in Python/Pydantic, and if it ever must be compiled, extend the Rust `hypergraph-rs` crate you already own rather than opening a third-language front.

**Why Textual for v1.0.** The decision is over-determined by facts already on the ground, not by Textual being the "best" TUI framework in the abstract:

- **It is already built and already contracted — but be honest about what is Textual-specific.** The *framework-bound* sunk cost is the client itself (`src/babylon/tui/`, 4,535 lines across 19 files), the KSBC `theme.py` (63 lines), and the Textual-Pilot test tier (`tests/unit/tui/`, 5,212 lines) — roughly **9.8k LOC** a rebuild would discard. The *framework-neutral* seam any client reuses is the projection layer (`src/babylon/projection/`, 11,426 lines across 57 files) and its tests (`tests/unit/projection/`, 10,658 lines), plus the archive integration tier (`tests/integration/archive/`, 1,195 lines) — which straddles, since the WO-50 pilot e2e drives Textual-Pilot while `test_vault_golden`/`test_county_e2e` assert the framework-neutral vault. All LOC re-counted on the working branch (S4 §1). Textual's genuine lead is therefore the ~9.8k framework-bound LOC — **not** the full estate; counting the ~11.4k projection layer and its ~10.7k of tests in Textual's column double-counts platform-neutral investment that survives a rebuild. The golden-vault ceremony (`blessed(archive-golden-vault)`, commit `00b7e488`) and the WO-50 pilot e2e (`tests/integration/archive/test_pilot_first_action.py`, 730 lines) already assert behavior against this client (S4 §1).
- **It is nearly free in the Nix closure you are already paying for.** The distribution story (ADR104) is built around *one* closure that already carries Python 3.12 + the full Babylon dependency set + a self-built Postgres + a 4.49 GB reference-DB derivation. Textual, `textual-image`, `dulwich`, and `jinja2` are pure-Python/small-binary wheels riding inside a Python environment that closure already builds, so the **packaging** delta is marginal (S4 §4). *Caveat on the one size figure:* the draft cited "≈209 MB" for "the Python closure," but its measurement method was not recorded, and 209 MB cannot be the *full* Python closure — pyarrow alone is ~100 MB, and scipy + duckdb + polars + psycopg exceed the rest. Read 209 MB, if anything, as the *marginal keel-wheel* set (textual/textual-image/dulwich/jinja2), and treat it as **[unverified — measure with `nix path-info -S` before quoting]**. "Marginal cost near zero" is a claim about **packaging**, not about **v1.0 effort** — the remaining game-loop program (§2, Appendix A) is separate and real.
- **The behavioral contract is partly wired to it, partly framework-neutral.** The T6 ruling makes the tutorial the BDD suite with *text* as the assertion medium. Only its **tier-1** assertions (exact greppable strings in `export_text` output + vault markdown) are framework-neutral and port to any client unchanged; **tier-2** (Pilot widget-tree queries) is Textual-Pilot-bound and **tier-3** (SVG snapshots) is Textual-bound. So "a rebuilt TUI is correct iff the tutorial suite passes" holds **only for tier-1** — a rebuild inherits the tier-1 text/markdown contract for free but must re-implement tiers 2–3 in its own harness (`ai/_inbox/t6-tutorial-bdd-ruling.md`, S4 §4). That the tier-1 contract lives in vault markdown + `export_text`, not Textual internals, is precisely what makes a future second client *possible*.
- **Disposability cuts *for* finishing it, not against.** The client-is-disposable ruling (Constitution II.8) is often read as "so it doesn't matter which we pick." The honest reading is the reverse: because the durable asset is the `observe()` seam + the golden vault, the cheapest path to a playable v1.0 is to finish the client that is already built against that seam, and keep the *option* to rebuild later cheap by keeping the contract in text. (No precise "percent wired" figure is claimed — the remaining v1.0 work is real and itemized in §2 and Appendix A.)

**Why not now-rebuilding in Neovim or Emacs.** The real cost is *maintenance surface, not bytes*. Both mean vendoring an entire second language/plugin ecosystem into the pinned closure (Lua + nvim's package manager, or Elisp + its archive), each with its own **version-pin, cache-population, and upgrade story** duplicating what ADR104 already solved once for Python. On raw download size the delta is *not* the argument: against ADR104's baseline (a 4.49 GB reference-DB derivation + a self-built Postgres + the full Python environment), a headless-nvim closure is hundreds of MB — a single-digit-percent size increase, not decisive. What *is* decisive is the *second standing pin/cache/version story* one solo maintainer must carry indefinitely **[judgment, grounded in S4 §4]**. Both also mean re-implementing the KSBC theme and dialog anatomy as bespoke highlight-groups/faces rather than porting the existing `theme.py` `Theme` object (S4 §5). And both would restart tiers 2–3 of the tutorial-BDD harness against a client that does not yet exist (only tier-1 text assertions carry over).

**The Neovim exception is real but narrow.** The hybrid — Python owns all state, drives a headless `nvim --embed` purely as a grid renderer/input surface over msgpack-RPC, isolated by `NVIM_APPNAME` — is the *designed* use of nvim, not a hack; nvim's own core team builds its default UI (`ui2`) on the same ext_ui/grid abstraction (S2 §2) — though see §7 for why that endorses the grid *model* more than the *external-RPC hop* the hybrid specifically depends on. If a Neovim-native second client is something you want *for yourself*, the vault-markdown seam makes it engine-safe to build post-1.0. But for the stated *casual Ubuntu player*, it carries genuine, dated-2026 risk (modal-editing adoption barrier, S2 §6; nvim-treesitter archived April 2026, S2 §1; pynvim thin maintenance, S2 §2; GNOME-Terminal image gap, S2 §4) and thin ROI over Textual. **Post-1.0, optional, BD-motivated — not v1.0.**

**Emacs** gives you genuinely impressive infrastructure (Transient/Casual action menus, buffer-as-live-view architecture, TTY child frames now merged to Emacs 31 — *but 31 is trunk/development as of 2026-07; the current release is 30.1*, S3 — live-redefinable Lisp image), but it is the heaviest *maintenance* tail of the three, the slowest cold-start without a persistent daemon, requires first-of-kind game-UI work (no game precedent exists on transient/buffers, S3 §1), and its baseline interaction vocabulary is hostile to a casual audience in ways which-key/Casual only partly fix (S3 §6). It is not a platform choice that earns a broad casual audience — any claim otherwise is marketing (S3, bottom line). **Not recommended for this project at any horizon** unless as a personal hobby client.

**OCaml vs Rust.** The suggestion to reach for OCaml for the algebra layer is not frivolous — its module system + functors + GADTs are a *genuinely better native fit* for encoding oppositions/constructors/derivation trees with compiler-enforced exhaustiveness over the legal cases (S1 §6), and dune's cram/expect tests match your golden-baseline doctrine (S1 §1). That is the one place the type-system argument is strong rather than aesthetic. But for **this repo**, the practical stack tilts hard to Rust: you already have working `PyO3 0.29` code in `hypergraph-rs` (S1 §4); the OCaml↔Python bridge (`pyml`, ~205 stars, single-maintainer, S1 §4) is hobby-scale next to PyO3's industrial one; Nix byte-pinning is one-lockfile-plus-hash in Rust vs a two-repo-pin opam-nix model (S1 §2); and — most load-bearing for an AI-agent-built project — the flagship coding benchmark (Aider Polyglot) scores Rust and not OCaml, so OCaml codegen quality has **no *Aider-Polyglot* signal** (absence from one benchmark is not absence of all signal — MultiPL-E and other multilingual evals may cover it; treat this as a benchmark fact with an assumed causal chain, not a proof, S1 §7). **Verdict: don't add OCaml.** Keep the algebra in Python/Pydantic for v1.0; if a compiled typed core becomes necessary, extend the existing Rust/PyO3 crate. Capture the exhaustiveness discipline OCaml would have bought you as *written predicate specs + property/exhaustiveness tests*, exactly as your own Fable-5 "behavioral contracts" doctrine already prescribes (CLAUDE.md). Because the algebra layer does not exist yet, deciding against OCaml now is fully reversible: "not now" is not "never" (§ Part II).

**Does Amendment AA change any verdict? No — it reinforces all of them.** The Windows-constraint pass (Part III; tagged W1–W7) stress-tested every recommendation against ADR108's binding post-1.0 Windows horizon (WSL2 = lane 1, native Windows = lane 2; pre-1.0 constitutionally shielded, `ADR108:decision`). Nothing moves. **Textual** ports: the WSL2 lane runs the Linux client verbatim, and on native Windows Terminal it works with exactly the caveats the glyph-floor ruling already anticipates — Sixel-only, no Kitty graphics protocol (microsoft/terminal#8389, open since 2020), open emoji-width bugs, an occasional 2026 mouse regression — so the glyph floor is the layer that carries, precisely as ADR108's own context predicts ("the ADR099 glyph floor is the rendering layer that ports") (W1, W4; `ADR108:context`). **The Neovim hybrid** ports the same way — Linux nvim verbatim under WSL2; on native Windows the nvim core is mature (0.12.1) but image.nvim is broken (Unix-only `ioctl`, 3rd/image.nvim#115) and WezTerm-Windows lacks the Kitty path (wezterm#5757), so its raster path degrades to the same glyph floor, no worse than Textual for a glyph-first game (W2, W4). **Emacs weakens further:** WSL2 is its strictly-easier lane, but the native lane piles on Windows-specific friction — an antivirus startup tax reported up to ~10s, MSYS2 build complexity, GUI-first testing bias, and no native terminal graphics even in the 2026 trunk — and its GUI-vs-TTY Windows parity is the one under-verified item in the whole pass (W3), pushing an already-not-recommended platform further down. **And OCaml's hit deepens:** Part II already recommended against it on Linux grounds; AA adds an axis on which Rust is Tier-1-mature on Windows while OCaml sits ~1–2 years behind with a live, unresolved package-ecosystem gap (opam crossed a real threshold at 2.2 but "not all packages are compatible" with Windows as of Jan 2026), so Rust's compiled-extension edge grows under the Windows horizon while OCaml's lone win — GADT exhaustiveness — is platform-independent and untouched (W7). The one-line Windows-impact disclosure duty AA imposes on load-bearing platform/language ADRs (`ADR108:decision` §3) is discharged for the client and language choices by Part III.

## Part I — The Interface Debate

### 1. What any client must satisfy

Before comparing platforms, fix what the target is. A Babylon client is not free to be anything; it must sit behind one seam and honor three fresh rulings.

**The seam — `observe()` + projections + the vault.** Constitution II.8 (Amendment V) defines a client as "a viewport into engine-computed state through the declared `observe()` projection contract… delivery is transport-independent… the terminal Archive client is [the browser client's] successor under this same contract" (`CONSTITUTION.md:354`, Article II §8 "Client as Presentation Layer"; the Amendment-V block that generalized it is at ~`CONSTITUTION.md:535`; S4 §3). Operationally there is no single `def observe()`; the contract is a family of per-subject `project_*` functions (`project_state`, `project_national`, `project_social_class`, `project_county`, …) each taking `(graph, world, tick)` and returning a frozen Pydantic view model, plus `projection/registry.py`'s declared SQL views, plus the player-intent write path (`verbs/submit.py::TurnSink` + `submit_verb()`), plus — critically — **the baked markdown vault**: one `.md` per subject under `<kind>/<id>.md`, plain committed-git markdown resolving `babylon://` wikilinks. That vault "is the actual nvim/emacs seam, since [it] is plain committed-git markdown, not Textual-specific" (S4 §3). Import-linter enforces the wall in both directions: `babylon.tui` may not import `babylon.engine`/`babylon.persistence`/`django`, and `babylon.projection` may not import `babylon.engine` (`pyproject.toml:631–642`, S4 §3). The **WO-37 structural-Protocol trick** (`tui/nav.py:10–14`) lets the composition root inject the real `BabylonMetaStore` while the client never imports persistence — the seam is structural on purpose (S4 §3). Any alternate client consumes exactly this surface; none of it is Textual-bound except `router.parse_babylon_uri` (logically transport-neutral) (S4 §3).

**Ruling 1 — the tutorial IS the BDD suite; text IS the assertion medium** (`ai/_inbox/t6-tutorial-bdd-ruling.md`, BD 2026-07-21). One frozen `TutorialStep` (given/when/then/anchor/predicate) is consumed three ways: a player-facing guided overlay, a headless CI run through the existing pytest+Pilot idiom, and developer-facing living documentation. Three strictly-ordered assertion tiers: (1) **semantic text** — Then-clauses assert exact greppable strings in `export_text`-rendered terminal text + vault markdown (this is what makes the golden vault the primary contract, and it is the *only* framework-neutral tier); (2) **structural Pilot widget-tree queries** (Textual-Pilot-bound); (3) **visual SVG snapshots** (Textual-bound), *aesthetic-only, never a behavioral gate* (S4 §4). A coverage sentinel must enumerate every player-facing option and assert each appears in ≥1 scenario. **Consequence for platform choice:** the durable, portable contract is tier-1 text and markdown, not framework widgets — so a platform is acceptable iff it can (a) emit deterministically assertable text and (b) be driven headlessly in CI, and a *rebuild* re-implements tiers 2–3 in its own harness.

**Ruling 2 — graph renders are outsourced; assert the SOURCE, display the RENDER** (`ai/_inbox/graph-render-lane-ruling.md`, BD 2026-07-21). Complex structural visuals are not hand-built widgets; they render graph-natively via `rustworkx.visualization` / XGI / (eventually) hypergraph-rs, displayed through the ADR099 kitty-raster lane with a **mandatory glyph floor** beneath. The behavioral/golden artifact is the *text input* to the renderer (DOT source, layout JSON) — never the rasterized bytes, which "aren't cross-version stable; DOT text is." The game must be fully playable glyph-only (e.g. over plain ssh) (S4 §4). **Consequence:** every platform must have a legible glyph-only floor; pixel graphics are a bonus lane, never a dependency.

**Ruling 3 — ADR104 nix-bootstrap installer** (accepted 2026-07-21). `install.sh` now installs Nix itself (pinned Determinate installer) then subscribes the signed R2 cache, carrying a game-managed Postgres cluster and the 4.49 GB reference-DB as a fixed-output derivation (S4 §4). **Consequence:** the distribution unit is one Nix closure. Anything already inside the Python environment is a marginal wheel; anything that is a *new language runtime + package manager* is a second closure surface whose real cost is the *standing* pin/cache/version story it adds, not its byte size against a 4.49 GB baseline **[judgment, S4 §4]**.

Three tests fall out of this for every candidate: **(T-text)** can it emit deterministic, greppable text and run headless in CI? **(T-floor)** does it have a legible glyph-only floor that survives GNOME Terminal and bare ssh? **(T-closure)** is it a marginal wheel, or a second language ecosystem adding a standing maintenance surface?

### 2. Textual (the incumbent)

**What it is as a platform.** A pure-Python, asyncio-driven TUI framework: one asyncio Task per widget, a CSS-like styling system with a `Theme` object of semantic variables, a compositor that diffs segments and uses a spatial-map grid for visibility, and an official SVG-diff snapshot test plugin (S5 §1). Current release 8.2.8 (2026-06-30); pinned here `textual>=8.2,<8.3` (`pyproject.toml:69`, S4 §2).

**Strengths.** Same language, same process, same closure — no second runtime, no RPC hop, lowest startup of the three (S5 §4). A real production track record (Posting, McGugan's Toad) proving it scales to serious tools (S5 §1). First-class snapshot testing (`pytest-textual-snapshot`) that maps directly onto the T6 text-assertion doctrine (S5 §1). A KSBC `Theme` already implemented and registered (`tui/theme.py`, S4 §5). It already satisfies the three **platform** tests — T-text, T-floor, T-closure — with **no new platform work**. (That is a claim about the *platform*, not about *v1.0*: the remaining game-loop-and-cutover program below is still real and unbuilt — see Appendix A.)

**Honest weaknesses.** (1) **Company is dead.** Textualize wrapped up May 2025; McGugan's own post-mortem calls Textual "a solution in search of a problem" — technically excellent, commercially unfundable (S5 §1). It is now single-maintainer-plus-community with no disclosed succession plan: a real bus-factor risk, though one already stress-tested for a year of healthy company-free releases (S5 §1). (2) **Major-version churn:** five major bumps in twelve months; most are narrow renames, but it makes `>=8.0,<9` pins meaningless as a stability signal — you must read the changelog line-by-line before upgrading, exactly the "always verify" tax a byte-reproducibility project resents (S5 §1). *(Note the tension: "a year of healthy company-free releases de-risks it" and "five majors in twelve months, pins are meaningless" describe the **same** release history. The honest net is a cadence that is active and community-sustained yet forces per-bump changelog diligence — reassuring on survival, taxing on upgrades.)* (3) **Perf ceilings:** built-in `DataTable` goes sluggish at a few hundred rows (third-party `textual-fastdatatable` exists to route around it); the GIL + asyncio-task-per-widget model is a hard ceiling under high-density UIs no compositor cleverness removes (S5 §1). For a turn-based simulation these ceilings are unlikely to bind **[judgment]**, but they are real.

**Packaging-in-Nix cost.** Effectively free: Textual/`textual-image`/`dulwich`/`jinja2` are wheels inside the Python environment the closure already builds and caches, with no second pin/cache story (S4 §4, S5 §4).

**Testing/BDD fit.** Best of the three. The WO-50 pilot e2e already runs real 30-system ticks in-process and asserts on projection/TUI seams (S4 §1); `pytest-textual-snapshot` + `export_text` are the native mechanisms the T6 ruling names (S4 §4, S5 §1). Caveat: SVG snapshots are aesthetic-tier only (never behavioral) per the ruling, and are known to re-bake wholesale on layout changes (MEMORY: ArchiveApp-layout re-bakes all goldens).

**KSBC theming surface.** Already done: `tui/theme.py` (63 lines) is a single source-of-truth `Theme` named `KSBC` with `primary=CRIMSON`, `accent=GOLD`, `background=FIELD`, plus custom variables, registered at `ArchiveApp.on_mount`; named hex constants are imported by other modules rather than re-hardcoded; a declared 256-color degraded fallback (`render.tiers.DEGRADED_256_PALETTE`) is drift-guarded by `test_design_bible_parity.py` (S4 §5). The dialog anatomy (hard-shadow plates, title-tab-breaks-border, inverse-video selection) is expressible in Textual CSS (S4 §5).

**Casual-player UX.** The best fit: no modal grammar, mouse works, standard app affordances. The remaining UX work is game design (verb plates, lobby flow), not platform-fighting.

**AI-agent codegen signal.** Python is the single best-benchmarked language for current coding models (Aider Polyglot, MultiPL-E, essentially every leaderboard covers it heavily) — for an AI-agent-built project the client's language being Python is a real, if second-order, velocity input (S1 §7 by analogy; see §6).

**Effort estimate.** *v1.0:* this is the already-charted remaining program — T4 builds the paced driver (`GameSession.advance_tick()` + `PausePredicate` seam exist; the driver itself does not, and `AsyncSimulationRunner` is dead-on-arrival, S4 §6), then T4-integration → BD GATE 3 → #241 cutover → T7 installer → T8. No new *platform* cost, only the planned game-loop-and-polish cost **[judgment: weeks, inside the existing program]**. *Post-1.0:* incremental — graph-render-lane integration + KSBC T8 aesthetic pass.

### 3. Neovim (BD-native; the hybrid candidate)

**What it is as a platform.** Two very different things, and conflating them is the central error. *(a) As an app host* (Lua plugins drawing UIs from buffers + extmarks + floating windows): a real, heterogeneous ecosystem — Neogit, octo.nvim, oil.nvim, render-markdown.nvim, even shipped games (S2 §1). *(b) As a remote-controlled renderer*: nvim ships **no built-in GUI**; it delegates all rendering to a UI client over msgpack-RPC. `nvim --embed --headless` is spawnable as a child process; a Python process can `nvim_ui_attach` and receive a `grid_line`/`grid_scroll`/`grid_cursor_goto`/`flush` redraw stream, owning all state and driving nvim as a pure grid + input surface (S2 §2). This is the designed use, endorsed *in spirit* by nvim's own core team building `ui2` on the same ext_ui/grid abstraction (S2 §2) — with the caveat (§7) that "ui2" appears to consume the grid model *in-process*, which validates the grid abstraction more than the external-RPC hop the hybrid proposes.

**Strengths.** For BD specifically: it is your native environment. The remote-UI pattern is architecturally sound and increasingly first-class. `NVIM_APPNAME` gives *fully solved* isolation — one env var retargets config/data/state/cache, so a `babylon-tui` profile never touches the player's real `~/.config/nvim` (S2 §3). Screen-state assertion is a real, battle-tested feature: `mini.test`'s `reference_screenshot()` diffs a captured child-process screenshot (including extmark highlights, virtual text, float position) against a golden, runnable headless in CI via `nvim-busted-action` (S2 §5) — this satisfies T-text/BDD honestly. Buffer-as-editable-UI (oil.nvim, render-markdown.nvim) is a legitimate match to a wiki-as-document client (S2 §1).

**Honest weaknesses.** (1) **Modal editing is a documented, dated-2026 adoption barrier** — week-one users measurably slower, "every intuitive shortcut doesn't work as expected." For a *casual* audience this is a real cost, not editor-snob FUD; it would demand a game-specific keymap layer that never asks the player to think in vim grammar, or an explicit narrowing of "casual" (S2 §6). Mouse support exists but is not default-on and is inconsistent over floating windows (S2 §6). (2) **nvim-treesitter was archived April 2026** after maintainer burnout — the single plugin that made "treesitter-highlighted buffer as UI" mainstream. Functionality moved into core 0.12's `vim.treesitter`, but query/parser packaging is now *your* problem in the Nix closure. Any markdown-as-wiki-UI plan must pin built-in 0.12+ and budget parser installation (S2 §1). (3) **pynvim is stable-but-thin:** 0.6.0 (2025-09), 33 open issues, ~1.7k stars, no roadmap — plan to be the one filing edge-case bugs (S2 §2). (4) **No published latency numbers** for the ext_ui redraw round-trip, and every keystroke pays an RPC serialize/IPC hop before your Python sees it (S2 §2, S5 §4). *That said, the hop is over a local unix socket — sub-millisecond in practice — and for a non-twitch, turn-based UI it is almost certainly imperceptible; this is the **same** turn-based charity extended to Textual's perf ceiling. Benchmark to confirm, but treat it as low risk, not a genuine blocker* **[judgment]**. (5) **`vim.ui.img` is 0.13-only, and 0.13 is nightly-only as of 2026-07-21** — the stdlib image path is not shippable against stable nvim; you're on plugin territory (image.nvim/snacks.image) or a vendored pinned nightly (S2 §3, §4).

**Packaging-in-Nix cost.** A second language ecosystem: a pinned nvim binary + config tree + pinned plugins (or the new built-in `vim.pack`) as their own closure component, with their own version-pin and cache story duplicating the Python one (S4 §4). On *size* a headless nvim build is hundreds of MB — a single-digit-percent delta against the 4.49 GB baseline, not the argument. The real cost is the *standing* second pin/cache/version story. Reproducible paths exist (`kickstart-nix.nvim`, home-manager `wrapRc=false`), but couple you to shipping/pinning nvim yourself (S2 §3).

**Testing/BDD fit.** Genuinely good via `mini.test` reference screenshots + `nvim-busted-action` (S2 §5) — but it is a *parallel* harness to the one T6 already built on pytest+Pilot; you inherit T6's tier-1 text assertions but re-implement tiers 2–3 against it, and screenshot determinism needs the same font/width/RNG pinning you already do for BLAS (S2 §5).

**KSBC theming surface.** New surface, not a port: custom highlight groups (`hl-BabylonCrimson`, …) in Lua/`nvim_set_hl` or a colorscheme file, *plus* a separate hand-rolled 256-color degrade table (nvim highlight groups don't auto-degrade the way Textual's `PaletteTier` does), *plus* bespoke overlay code for the dialog anatomy — nvim has no first-class analog to Textual's `Theme`/CSS variables (S4 §5).

**Casual-player UX.** The weakest of the modal/non-modal axis for a broad audience (S2 §6), best for a BD-native audience of one.

**AI-agent codegen signal.** A Neovim client's implementation language is **Lua**, which is thinly covered on coding leaderboards relative to Python — a real, if second-order, drag on agent codegen velocity for an AI-agent-built project (see §6).

**Effort estimate.** *v1.0:* not viable as the primary — it restarts the client, the theme, and tiers 2–3 of the BDD harness. *Post-1.0 second client:* a real multi-week-to-months project **[judgment]**: a pynvim renderer consuming vault markdown + projection view-models, a game keymap layer, KSBC highlight groups + degrade table, dialog overlays, verb-submission + verb-plate + pause-UI client code (see §7), and a `mini.test` screenshot suite. The seam makes it *engine-safe* (never imports engine/persistence — vault markdown is the interface); disposability *sanctions the build cost* — but note it does **not** cover the *standing* two-client maintenance tax (Appendix A). Worth it iff you want a Neovim-native client for yourself.

### 4. Emacs

**What it is as a platform.** A live Lisp image with a buffer-centric application model. The relevant modern app patterns are real and mature: **Transient** (Magit's factored-out menu library; prefix/suffix/infix maps cleanly onto a turn-action menu), **Casual** (a suite proving Transient generalizes to non-git modes), **elfeed/notmuch/org-roam** (buffer-as-live-filtered-view over a store — the closest existing analog to wiki-as-game-client) (S3 §1).

**Strengths.** A genuinely richer *live/introspectable* dev environment than either alternative — redefine a function while the game runs, no restart (S3 §5). Buffer-as-document-store precedent (org-roam is a SQLite-backed backlink wiki in production) is stronger than nvim's plugin ecosystem for this shape (S3 §1). **In 2026:** TTY child frames merged to Emacs 31 — floating popup panels (inventory, dialogs, tooltips) in plain terminal Emacs, undercutting "TUI Emacs can't do floating UI" (S3 §2, "surprise"). *But apply the same "is it in a stable release?" test used on nvim's 0.13-nightly `vim.ui.img`: Emacs 31 is **development/trunk** as of 2026-07 — the current release is 30.1 (S3, Phoronix "Emacs 30.1 Released") — so TTY child frames are a **trunk** capability, not a shipped-stable strength.* Headless testing is solid and battle-tested: `emacs --batch` + ERT, `(should (equal (buffer-string) expected))` is idiomatic; `emacs-overlay` gives strong reproducible source/config distribution (S3 §3, §4).

**Honest weaknesses.** (1) **No game precedent.** Transient/Casual/elfeed/org-roam are all *productivity* patterns; no widely-used game engine exists on transient — this is first-of-kind UI work, not reuse (S3 §1). (2) **TTY child frames are frame compositing, not pixel graphics** — mainline Emacs still has *zero* native Sixel/Kitty support even in the 2026 trunk; terminal images require third-party packages (`kitty-graphics.el`) (S3 §2, "surprise"). Child frames also carry a live UX-engineering tax even on GUI (focus-stealing broken on Wayland/GTK, flicker, coordinate-math corruption — each package re-solves the workarounds independently) (S3 §2). (3) **native-comp + Nix has open friction** — missing `.eln` files for built-ins, and native-comp's "compile lazily on first use" design fights Nix's immutable read-only store; you accept first-run JIT latency or extra plumbing (S3 §4). (4) **Startup** is structurally higher than nvim's regime; the mitigation is a persistent `emacsclient` daemon — a real operational commitment a one-shot game shouldn't need (S3 §4, S5 §4). (5) **Elisp is single-threaded and dynamically typed** with dynamic-scoping-by-default footguns; native-comp narrows but doesn't close the LuaJIT gap (S3 §5). (6) **Casual-player UX is the worst of the three:** chord-based commands with zero visual affordance and a buffer/frame/window vocabulary unfamiliar to any non-Emacs user; which-key/Casual help "someone willing to read a menu" but don't solve the baseline-vocabulary problem — "not a platform choice that gets you a broad casual audience for free… any claim otherwise would be marketing" (S3 §6, bottom line).

**Packaging-in-Nix cost.** Heaviest *maintenance* tail of the three by dependency-tree shape — a GUI/toolkit Emacs drags in GTK/Cairo/image libs/tree-sitter/etc.; even source/config reproducibility is strong via `emacs-overlay`, but the native-comp compiled-artifact caching does not play cleanly with the immutable store (S3 §4, S5 §4).

**Testing/BDD fit.** Solid on its own terms (batch+ERT, buffer-string assertions) but, like nvim, a parallel harness that inherits only T6's tier-1 text assertions and restarts tiers 2–3 (S3 §3).

**KSBC theming surface.** New surface: custom `defface` declarations in an elisp theme + bespoke overlay code for dialog anatomy; no first-class `Theme`/CSS-variable analog (S4 §5).

**AI-agent codegen signal.** An Emacs client's implementation language is **Elisp**, which is essentially absent from coding leaderboards — the weakest of the three for agent codegen velocity on an AI-agent-built project (see §6).

**Casual-player UX.** Realistically pitched at an audience that already tolerates Emacs (S3 §6).

**Effort estimate.** *v1.0:* not viable (first-of-kind game UI + restarted contract + heaviest maintenance tail). *Post-1.0:* not recommended for the stated audience at any horizon **[judgment]**; defensible only as a personal/hobby client for an Emacs-native player, where the live-Lisp DX is the whole reward.

### 5. Who rasterizes what — the correction

This must be said precisely, because it recurs across all three platforms and directly determines the glyph-floor ruling. **There are three layers, and only one of them ever touches a pixel:**

1. **The terminal emulator** (kitty, Ghostty, WezTerm, Konsole, iTerm2, and Ubuntu's default **GNOME Terminal/VTE**) is the *only* layer that turns anything into a framebuffer. It owns the GPU/software rasterizer, decodes the PNG/RGBA payload a graphics-protocol handshake hands it, and paints both the text cells *and* any inline image into the window it draws on the display server (S5 §3, S2 §7). For the Kitty protocol specifically, the client sends base64-chunked RGB/PNG data over an APC escape sequence and the terminal decodes/composites/draws it. **Placement has two modes**: the *default* is **cursor-anchored** — the image is pinned at the cursor position and does **not** scroll with the text — while the optional **Unicode-placeholder** mode encodes the image's cells with `U+10EEEE` plus diacritic-encoded row/column so the image *does* scroll like ordinary cell text. That placeholder mode is the specific mechanism a scrolling TUI uses to make an image behave like text; it is not the general placement mechanism (S2 §4).

2. **The TUI framework** (Textual, Ratatui, and by exact analogy nvim's grid model) *never rasterizes*. It computes which characters + SGR styling belong in each cell, or — for an image — which protocol-specific escape blob to hand the PTY, and writes that byte stream to stdout. Its entire output is text; it is blind to whether the terminal on the other end can even parse the graphics escapes it emits. Textual's `textual-image` may *decode* an image to raw pixels before handing the terminal a protocol blob, but on the KGP path it does not do the on-screen rasterization — that sits squarely with the terminal (S5 §3, S2 §7). *One nuance for exactness:* in its **half-block/Unicode fallback** (terminals with no graphics protocol), `textual-image` **does** sample the image down into colored character cells — a rasterization-*to-grid* — but the output is still text (colored cells), not a framebuffer blit. So the glyph-floor path is itself a grid rasterization; the KGP path is not. Neovim's TUI (`src/nvim/tui/tui.c`) is itself just one UI client translating `grid_line` events into ANSI/UTF-8 for the PTY — it never touches pixels either (S2 §7). **The cell grid is the interface, not the output.**

3. **GUI-hosted editors that run their own window** — **Neovide** and **GUI Emacs** (GTK/Cocoa, not `--nw`) — are a *third* category: they do not run inside anyone else's terminal at all. They open their own native window and rasterize glyphs themselves. But their **process architecture differs sharply**, and treating them as identical (as the draft once did) is wrong:
   - **Neovide** is an *external* GUI process attached to a *separate* `nvim` core over **msgpack-RPC** (ext_linegrid): nvim computes the grid and ships `grid_line`/cursor events across the RPC channel; Neovide receives them and rasterizes via wgpu/OpenGL. This is a genuine core/UI **process split with an RPC hop** — the same remote-UI architecture the Neovim hybrid in §3/§7 proposes.
   - **GUI Emacs** is the opposite: a **monolithic single process**. Its C core drives redisplay directly through the toolkit (Cairo/GTK) *in-process* — there is **no** core/UI process split and **no** RPC channel. "GUI Emacs" and "terminal Emacs" are one and the same process choosing a graphical vs a terminal display back-end, not a client attached to a server.
   Both open their own window (no intervening terminal emulator, no PTY-level graphics negotiation, S5 §3, S2 §7). This is why the "images" story is completely different for GUI Emacs (native libpng/librsvg, `svg.el`, real game precedent like `svg-2048`, S3 §2) than for terminal Emacs (third-party Sixel/Kitty packages only).

**The load-bearing consequence for *this* project:** GNOME Terminal/VTE — the terminal a casual Ubuntu player actually has — supports *neither* the Kitty graphics protocol *nor* (reliably) Sixel out of the box (S2 §4, S5 §2). A casual Ubuntu player using the stock terminal will see **nothing** where an image should be unless they additionally install kitty/WezTerm/foot/Ghostty. This is exactly why the graph-render-lane ruling makes the **glyph floor mandatory** and rasters a bonus lane: "assert the SOURCE (DOT text), display the RENDER" — the game must be fully playable glyph-only (S4 §4). tmux makes it worse (KGP silently discarded without `allow-passthrough`; the native kitty-image tmux branch exists but is unmerged as of mid-2026, S5 §2). The correction is not academic: it validates the ruling and it applies identically to whichever framework you pick.

### 7. The hybrid option, taken seriously: second client vs rebuilding now

The interesting long-horizon question is not "which framework" but "should the *second* client be Neovim-as-renderer, and when?"

**The architecture is real, not a hack.** Python owns 100% of state (it already does — `GameSession` is a UI-agnostic composition root; `VaultPageSource` is a plain type alias kept precisely so the module "does not pull in Textual merely to describe [its] return shape," S4 §6). A headless `nvim --embed --headless` is attached over msgpack-RPC; Python calls `nvim_ui_attach(w, h, ext_linegrid=True)` and consumes the `redraw` stream, treating nvim as a pure grid renderer + input source. `NVIM_APPNAME=babylon-tui` fully isolates it from the player's real config (S2 §2,§3). `ext_multigrid`/`ext_popupmenu`/`ext_cmdline` let the Python UI own floats and widgets rather than nvim drawing them.

*On the ui2 endorsement — precision matters.* nvim's own `ui2` is built on the same ext_ui/grid abstraction (S2 §2). But "uses internally" cuts against the strong reading: if `ui2` consumes the grid model **in-process** (rather than running as an out-of-process RPC UI client), it endorses the **grid abstraction**, not specifically the **external-msgpack-RPC hop** the hybrid depends on. The research streams did not pin down ui2's process boundary. So treat ui2 as validating the grid model *for certain*, and the external-RPC-client architecture *only if* it runs out-of-process — a weaker endorsement than "the core team ships exactly our architecture." (Neovide, by contrast, *is* an out-of-process RPC UI client, §5 — so the external-RPC pattern is validated by Neovide regardless; ui2 just doesn't add to that the way the draft implied.)

**Why it is a *second* client, not the first.** Three reasons converge. (1) The disposability ruling means the durable asset is the vault-markdown + projection seam, which is already text and already tested; a second renderer consumes it without touching the engine (S4 §3). (2) The T6 BDD contract is text/markdown **at tier-1 only**: a Neovim renderer that faithfully displays the same vault pages passes the same **tier-1** semantic-text assertions for free — but *only* tier-1. Tier-2 (Pilot widget-tree) and tier-3 (SVG) are Textual-bound and must be re-implemented in a parallel `mini.test` screenshot harness (§3, §5). So the precise claim is: a rebuilt TUI is correct **iff its tier-1 text/markdown assertions pass *and* it re-implements tiers 2–3** — not "iff the tutorial suite passes" full stop (S4 §4). Moreover, "faithfully displays the vault pages" understates the job: the seam also includes the **player-intent write path** (`verbs/submit.py::TurnSink` + `submit_verb`), the **verb plates**, and the **live-loop pause UI**, none of which live in the read-only vault markdown — a second client must re-implement all three as new client code, not merely render markdown (S4 §3, §6). (3) You are Neovim-native; a Neovim client has real value *for you* even if its casual-player ROI is thin.

**Why not rebuild now.** Rebuilding now converts a solved v1.0 into an unsolved one: it restarts the client (4,535 LOC), the theme (`theme.py` → highlight groups + degrade table + dialog overlays, S4 §5), and tiers 2–3 of the BDD harness (Pilot → `mini.test` screenshots, S2 §5), and it adds a second language ecosystem — with its standing pin/cache/version tax — to the closure. All to serve a casual audience for whom the modal-editing barrier (S2 §6) and the GNOME-Terminal image gap (S2 §4) are *net negatives*, while carrying the treesitter-archival (S2 §1) and pynvim-thinness (S2 §2) risks. That trade is strongly negative before 1.0 and merely optional after **[judgment]**.

**The honest hybrid recommendation.** Ship Textual for 1.0. *After* 1.0, if you want it, build the Neovim renderer as a second client against the frozen vault+projection seam, reusing the tutorial suite's **tier-1** text assertions as its acceptance gate and re-implementing tiers 2–3, the verb write-path, verb plates, and the pause UI as new client code. Keep it strictly a renderer/input surface (Python owns state), keep it `NVIM_APPNAME`-isolated, pin nvim + parsers explicitly in the closure, and benchmark the redraw round-trip before committing to a live-loop cadence — though a local-socket RPC hop is very likely imperceptible for a turn-based UI, so treat that benchmark as due diligence, not an expected blocker (S2 §2). Do not treat it as replacing Textual — treat it as the disposability ruling working as intended: two disposable viewports over one durable seam. And weigh the *standing* two-client maintenance surface (Appendix A) before committing, not just the build cost.

## Part II — OCaml vs Rust for the Algebra / Renderer Layer

**Scope reminder:** the OCaml suggestion is for the *Lawverian algebra layer* — a typed encoding of oppositions, constructors, and derivation trees (co-Heyting boundary operators, contradiction families) — **not** the engine and **not** the client. The renderer question (rustworkx/XGI/hypergraph-rs → DOT → raster) is already ruled and is Python-orchestrated with a Rust option already in-tree, so this Part focuses on the algebra layer, then folds in the renderer.

### 1. Why OCaml was suggested — and it is not frivolous

The case for OCaml here is the one place in the whole comparison where the type-system argument is *strong rather than aesthetic* (S1 §6):

- **GADTs give compiler-enforced exhaustiveness over the *legal* cases only.** OCaml's checker special-cases GADT wildcards — it splits a wildcard into concrete constructors and type-checks whether each is even *inhabited* before demanding it be handled (S1 §6). That is precisely what you want for a `Contradiction` GADT indexed by opposition-kind, where only well-typed derivation steps type-check at all and the compiler proves you've covered every reachable case. This is a direct mechanization of the "anti-permissiveness" you want in a dialectic algebra.
- **Module functors are the native analog of parameterized categorical structure.** OCaml signatures + functors "package category-theoretic functors with abstract type declarations and an `fmap`," playing the role Haskell's type classes do — i.e. "opposition algebras parameterized over a base category" is a native construct, not an emulation (S1 §6).
- **Rust has no GADTs.** The Rust community's own writeups (blog.csongor.co.uk, the internals thread literally titled "Haskell style data constructors (GADTs) for rust enums") confirm this is simulated via traits + associated types + phantom markers — "clever workaround," not native fit, and it does *not* give GADT-style exhaustiveness over type-refined cases (S1 §6).
- **dune's cram/expect tests** match your golden-baseline doctrine unusually well — snapshot tests of toplevel output are first-class, something Rust bolts on via `insta`/`trycmd` (S1 §1).
- **Honest caveat even here:** the categorical-semantics literature ("GADTs, Functoriality, Parametricity: Pick Two") documents a genuine tension — GADTs' case-analysis power does *not* automatically preserve functoriality (S1 §6). So "GADT = free categorical structure" is itself an over-claim; the win is exhaustiveness and abstract-type packaging, not a free functor.

If the algebra layer were being built in a vacuum, OCaml would be a defensible, even strong, choice for it.

### 2. The honest practicality comparison, for *this* repo

It is not being built in a vacuum. Against the repo's binding facts, the practical stack tilts to Rust on every axis except the type-system one:

- **Python interop — the sharpest asymmetry.** This repo *already has working PyO3 code* on the current release line: `hypergraph-rs/crates/hypergraph-rs-python/Cargo.toml` depends on `pyo3 = "0.29"` (verified, S1 §4). PyO3 is industrial: 15.9k stars, 105 releases, free-threaded 3.14 support, the maturin "profile → `maturin develop` → ship a wheel" pipeline (S1 §4). The OCaml equivalent, `pyml`, is ~205 stars, single/small-maintainer, most-recent activity a compatibility-patch release, no maturin-equivalent (S1 §4). Since Python is *literally the core layer* (Pydantic models, GameDefines), the choice is not "OCaml has interop" — it's "OCaml has a hobby-scale interop library where Rust has an industrial one you have already used" (S1 §4). This alone is a heavy thumb on the scale.
- **Nix byte-reproducibility.** Rust is one lockfile + one `cargoHash` (with the one known gotcha: re-pin the hash on version bump). OCaml is opam-nix's *two-repository-pin* model — nixpkgs rev *and* opam-repository rev must agree — layered under a translation tool, with legitimate competing tools (opam-nix vs opam2nix vs esy) that are themselves a maturity signal against it (S1 §2). For a project whose whole distribution story (ADR104) is "one pinned closure," Rust is less painful by a real margin (S1 §2).
- **AI-agent velocity — load-bearing for how this repo is built.** The flagship coding benchmark, Aider's Polyglot (225 exercises), covers C++/Go/Java/JS/Python/**Rust** — *not OCaml* (verified against the benchmark's own language list, S1 §7). For an AI-agent-heavy workflow, this means every Rust snippet an agent emits comes from a model benchmarked and presumably tuned against Rust. **But be precise about the strength of this claim:** OCaml is absent from *Aider Polyglot specifically*; that is not absence of *all* signal (multilingual evals such as MultiPL-E may include it). Read it as a first-order velocity **consideration** with a solid benchmark fact but an *assumed* causal chain (benchmark presence → agent codegen quality → project velocity) — not a proven risk (S1 §7).
- **Ecosystem/docs/portability.** crates.io ≈ 200k vs opam ≈ 4.5k (~40–50×); Real World OCaml's current edition (2022) is now ~4 years stale against OCaml 5.3–5.5 (S1 §7); the "Setting up a new OCaml environment in 2026 is not yet a fluid experience" thread is the most honest current data point on onboarding friction (S1 §1); Rust's musl/ARM/`cross` tooling is more turnkey (S1 §3). None decisive alone; cumulatively one-directional.
- **Determinism.** Rust's no-GC model forecloses a whole class of "did the collector's timing leak into my tick hash" bugs *by construction* — a narrow but real edge for a Constitution that treats deterministic-hash as inviolable. OCaml's GC does not compromise *logical* determinism as long as no GC-observable state leaks in (same constraint Python's GC already imposes) — so this is a Rust plus, not an OCaml disqualifier (S1 §5). The float/libm caveat (transcendentals like exp/log not cross-platform reproducible) is *identical* in both languages and identical to your current Python — it needs the same tolerance-policy discipline regardless, exactly as your Fable-5 doctrine already states (S1 §5).
- **Where OCaml genuinely wins** remains: GADTs/functors as native categorical encoding, and dune's cram/expect tests (S1 §6, §1). Toss-ups on both sides: concurrency-runtime maturity (Eio ~2yr vs Rust's fragmented-but-hardened async) and Windows (moot for a Linux-only target) (S1 §5).

### 3. Verdict for this repo

**Do not adopt OCaml. Keep the Lawverian algebra in Python/Pydantic for v1.0; if it must become a compiled typed core later, extend the existing Rust `hypergraph-rs` crate via PyO3 rather than opening an OCaml front.** The reasoning, weighed honestly against the one real OCaml advantage:

The decision is a genuine tension because OCaml's single win lands *exactly on the layer under discussion* — this is not a case where OCaml's strength is irrelevant. But three facts about *this* repo override it. First, the algebra layer is small, load-bearing, and correctness-critical — precisely the kind of code where the *cost* of a new language (a ~200-star interop library, a two-repo Nix pin, a benchmark-thin AI-codegen story, a second GC/toolchain to reason about, a third language for every future agent to context-switch into) is paid on every single edit, while the *benefit* (compiler-checked exhaustiveness) can be recovered another way. Second, you already own the cheaper compiled path: `hypergraph-rs` is real, on current PyO3, and paused — not absent (S1 §4). **Be precise about *what* is reused, though:** `hypergraph-rs` is a graph-*substrate* crate (petgraph-backed hypergraph ops), **not** a Lawverian-algebra crate — co-Heyting boundary operators and contradiction GADTs are a different domain. What you reuse is the **PyO3/maturin/Nix compiled-extension pipeline and the precedent you already paid for**, not algebra-relevant code; "extend the crate" means "reuse the pipeline," not "the crate is a natural host for the algebra." Introducing OCaml would instead mean maintaining *three* languages (Python + the Rust you already have + OCaml) in one pinned closure, against an AI-agent workflow with no Aider-Polyglot OCaml signal to lean on. Third — the reconciliation with your own doctrine — the exhaustiveness discipline OCaml would enforce *at the compiler* is exactly what your Fable-5 "behavioral contracts" doctrine already says to encode as **written predicate specs + property/exhaustiveness tests** (CLAUDE.md: "grow deliberately… golden baselines, property-based laws, written predicate specs"). A Python `match` over a frozen-Pydantic constructor family, guarded by an exhaustiveness test and a property law per opposition, gets you most of the GADT safety in the language your core already speaks — and it is *language-agnostic to the byte*, which the rewrite test prefers over a type-checker's guarantee that dies with the implementation **[judgment]**.

**Reversibility.** Because the algebra layer *does not exist yet* (v1.0 keeps it in Python), deciding against OCaml now is fully reversible: if the layer later materializes and Python proves inadequate for it, OCaml — or, more likely given the toolchain you already own, Rust — can be added then. "Not now" is emphatically not "never"; the option value costs nothing to preserve **[judgment]**.

**On the renderer half of the question:** it is already resolved and does not favor OCaml at all. The graph-render lane is ruled to be `rustworkx.visualization`/XGI/eventually `hypergraph-rs`, orchestrated from Python, asserting DOT text and displaying rasters through the kitty lane with a glyph floor (S4 §4). Rust's TUI/graphics ecosystem is deeper and more battle-tested than OCaml's (ratatui/crossterm vs the young sub-1.0 Minttea / documentation-poor lambda-term, S1 §8), but that comparison is moot here because the renderer is Python-orchestrated with an in-tree Rust option, not an OCaml candidate.

**Net:** the OCaml suggestion identified a real property worth wanting (exhaustive, type-refined derivation trees). The right response is to *capture that property as a testing/spec discipline in Python*, keep Rust as the sole compiled-extension language on the strength of the PyO3 precedent you already paid for, and not pay the three-language tax for a compiler guarantee your own doctrine already tells you to make language-agnostic.

## Part III — The Windows Constraint (Amendment AA)

Parts I–II reasoned on a Linux-only target. **Amendment AA (ADR108, Constitution v2.14.0→v2.15.0, accepted 2026-07-21)** changes the horizon, not the target: it makes Windows compatibility a *binding post-1.0.0* requirement while *shielding* pre-1.0 development from it entirely. This Part is the section ADR108 names by path — it records that "the in-flight UI-debate report gains a Windows-constraint section weighting Textual/Neovim/Emacs and Rust-vs-OCaml under AA" (`ADR108:consequences`). Its job is to verify, against primary sources, the "known platform facts… to be verified in depth by reports/user-interface-debate.md, in flight" that the ADR listed but did not itself prove (`ADR108:context`). Claims are tagged **W1–W7** to the seven verification areas of the Windows pass (sources in Appendix B).

### The constitutional rule

- **Binding post-1.0.0, two ordered lanes.** Windows compatibility is a *required post-1.0.0 portability milestone*, activated by BD ruling once the game concept is proven. **Lane 1 = WSL2** — the existing flake channel unchanged, promoted from "documented workaround" to a supported path plus a doctor preflight. **Lane 2 = native Windows** — the 2.0.0 release, a *separate non-Nix packaging story* whose channel ADR is deferred to activation (`ADR108:decision` §1).
- **Pre-1.0.0 is constitutionally shielded.** No Windows CI legs, no Windows-conditional code paths, no speculative cross-platform abstractions, no v1.0.0 DoD item referencing Windows; any pre-1.0 Windows work requires amending AA first (`ADR108:decision` §2). So this Part must not induce a single line of pre-1.0 Windows work — it is a *horizon-scan*, not a work order.
- **The one pre-1.0 obligation is disclosure, never a gate.** Load-bearing platform/language/dependency ADRs carry a one-line Windows-impact note and, all else equal, prefer the non-foreclosing option. Two seams are already named as carrying that optionality: the unified DSN resolver's TCP-loopback fallback point, and the ADR099 glyph-floor playability discipline (`ADR108:decision` §3).

**The framing correction the research forces, that ADR108's context does not yet spell out:** *WSL2 does not itself grant any terminal-graphics capability.* Rendering is gated by whichever Windows-side terminal *fronts* the WSL2 session, so the native-Windows terminal-graphics gaps below apply **verbatim in the WSL2 lane too** — WSLg (X11/Wayland GUI passthrough) is orthogonal to terminal-graphics protocols and does not close them (W1, W4, W5). This matters because it means the glyph floor, not the raster lane, is what actually ports across *both* Windows lanes — reinforcing, not complicating, the ADR099 discipline AA leans on.

### 1. Per-platform Windows matrix (native lane + WSL2 fidelity)

**Textual.** *Native:* true color works on modern Windows Terminal (WT); the legacy Windows Console/conhost is still capped at 16 colors, and Textual explicitly lists WT alongside iTerm2/Alacritty/Kitty/VTE as a working target (W1). Mouse has a long-standing class of ConPTY bugs that swallow/mis-scale mouse escape sequences (microsoft/terminal#376, #3364, #4973), plus a **2026-specific regression**: Windows 11 25H2 build 26200.8655 (June 2026 cumulative update KB5094126/KB5094135) made mouse clicks stop working system-wide while a VT-mouse-mode TUI runs in WT — workaround is running under conhost (W1). Emoji/CJK width is an **unresolved multi-year bug class** (microsoft/terminal #674, #900, #8970, #16852, #17342 — the last a live 2026-era report): wcwidth mismatches and ignored VS15/VS16 presentation selectors, still open mid-2026 (W1). For images, `textual-image` supports Kitty's TGP and Sixel with a Unicode half-block fallback; **WT supports Sixel since v1.22 (preview)/v1.23 (stable) but has no Kitty Graphics Protocol** (microsoft/terminal#8389, open since Nov 2020, Backlog/P3/unassigned; the 2024 duplicate #17309 was closed unimplemented), and Sixel-in-Textual is reported low-performance — so on native WT `textual-image` gets *Sixel-or-Unicode-fallback, never TGP* (W1). *WSL2 fidelity:* rendering happens in the Windows-side terminal hosting the WSL2 shell, so **every WT-specific gap above (no Kitty TGP, emoji-width bugs, mouse regressions) applies verbatim** — WSL2 grants no different terminal-graphics capability (W1, W5).

**Neovim.** *Native:* mature build quality — official `nvim-win64.zip` releases, no admin required, current stable 0.12.1 (April 6 2026), Windows 8+ (7 unsupported); WT is the recommended pairing, with true-color/undercurl gated on WT's ConPTY/OpenConsole plumbing (W2). **image.nvim is broken on native Windows** — it crashes with "cannot resolve symbol 'ioctl'" because `ioctl()` is a Unix-only syscall used for its terminal-size query (3rd/image.nvim#115, opened Jan 2024, closed with no fix). WezTerm does not rescue it: WezTerm's **Windows build implements only the iTerm2 image protocol, not Kitty's** (wezterm#5757) — the Kitty path that works on WezTerm-for-Linux does not exist on WezTerm-for-Windows (W2). `pynvim` has no Windows-specific blockers — it connects over named pipes on Windows (the native analog of the Unix-socket transport), functionally fine, just a different transport string (W2). *WSL2 fidelity:* Neovim + image.nvim + Kitty TGP work exactly as on native Linux *inside* the WSL2 shell, but only if the Windows-side terminal forwards the graphics escapes — which WT does not for Kitty TGP — so even in the WSL2 lane image.nvim's Kitty path is gated by the fronting terminal, not by WSL2 (W2, W4, W5).

**Emacs.** *Native:* native-comp (libgccjit) is available on Windows via the official installer (29.3+) and a documented MSYS2/mingw-w64 self-build path (W3). Startup carries a Windows-specific tax: on-demand native compilation adds first-run latency until the `.eln` cache warms, and — separately — Windows Defender/antivirus scanning has been reported to add up to ~10s to Emacs startup on Windows, with no Linux/macOS equivalent (W3). The GUI build (`runemacs.exe`) is the primary, best-tested native path; `emacs -nw` (TTY) works but is less documented/tested on Windows specifically (W3). **Honest caveat (the one under-verified item in this pass):** source coverage for a canonical, up-to-date (2026) GUI-vs-TTY performance comparison on Windows was thin — most material dates from the Emacs-28-era native-comp rollout (2020–2022), with only the antivirus-overhead datapoint clearly 2026-adjacent. Treat the conclusion "functional but with Windows-specific friction (AV overhead, MSYS2 build complexity, GUI-first testing bias)" as directionally solid but flag it for a follow-up BD verification pass *if Emacs ever becomes a serious contender* — this is the one W-item not verified to primary-source depth (W3, honest caveat) **[judgment]**. *WSL2 fidelity:* Emacs runs as ordinary Linux Emacs inside WSL2 (TTY, or GUI via WSLg) — none of the native-comp/antivirus friction applies; the strictly-easier lane for Emacs (W3, W5).

### 2. Terminal-graphics on Windows — "where do images work" matrix

| Terminal (native Windows) | Sixel | Kitty Graphics Protocol |
|---|---|---|
| Windows Terminal | **YES** (v1.22 preview / v1.23 stable+) | **NO** — open request since 2020, Backlog/P3, no roadmap (microsoft/terminal#8389) |
| mintty | **YES** (v2.6.0+) | **NO** |
| WezTerm (Windows build) | listed supported | **Effectively broken** — Windows build completes only the iTerm2 path; the Kitty path that works on WezTerm-Linux does not (wezterm#5757) |
| kitty | N/A | N/A — **no native Windows port at all** (kovidgoyal/kitty#4069); users are directed to run it inside WSL2 |
| Ghostty | N/A | N/A — **Linux/macOS only as of 2026**, no committed Windows timeline; only unofficial community forks exist |
| ConEmu / Windows Console / PuTTY / MobaXterm | **NO** | **NO** |

**Bottom line (Part I §5's "still absent?" ask, answered for Windows):** as of 2026, Kitty's Graphics Protocol has **no working native-Windows path in any mainstream terminal** — not WT (never implemented), not WezTerm-on-Windows (implemented on Linux, broken on Windows), not kitty (no Windows port), not Ghostty (no Windows port). The **only** natively-working image path on Windows is **Sixel**, and only in WT (≥1.23) or mintty; everything else (image.nvim, `textual-image`'s TGP path) falls back to Unicode-block rendering or fails (W4). *WSL2 lane:* identical to native — the gap lives in the Windows-side terminal emulator, not the Linux guest, so hosting a WSL2 shell in WT or WezTerm-for-Windows inherits the exact same Kitty-protocol absence (W4, W5). **This directly vindicates the graph-render-lane ruling's mandatory glyph floor** (Part I §5): the same reasoning that makes a stock-GNOME-Terminal Ubuntu player see *nothing* where an image should be reproduces on Windows across *both* lanes — the glyph floor is what ports, the raster lane is a bonus Windows mostly cannot honor natively, exactly as `ADR108:context` asserts ("the ADR099 glyph floor is the rendering layer that ports").

### 3. WSL2 lane fidelity — what degrades vs native Linux

- **Graphics protocols through the WSL boundary:** *not* solved by WSL2/WSLg — gated by whichever Windows-side terminal hosts the session (items 1–2 above). WSLg is GUI-app (X11/Wayland) passthrough, orthogonal to terminal-graphics protocols (W5).
- **Clipboard:** functional but fragile — WSL's bridge depends on Windows executables (`clip.exe`/`powershell.exe`) being resolvable on the WSL-side PATH; a Jan-2026-reported failure mode breaks clipboard specifically when those aren't found in PATH inside the guest (opencode#9999). A config-sensitive rough edge, not a hard blocker (W5).
- **Unix sockets:** WSL2 runs a genuine (lightweight-VM) Linux kernel, so `AF_UNIX` works completely natively inside the WSL2 filesystem/namespace — a real structural advantage over native Windows for anything unix-socket-first (e.g. Postgres's traditional local-connection default; see §4 for the more nuanced native answer) (W5).
- **Postgres in WSL2:** fully functional; the make-or-break variable is filesystem placement — data directories on the WSL2 **ext4** fs perform at near-native Linux speed, while data through `/mnt/c/...` (the 9p bridge) is markedly slower. The consensus "good enough for dev/test, an impairment for production" matches Babylon's dev-box use case *as long as PGDATA stays inside the Linux fs* (W5).
- **Ollama GPU from WSL2:** strong — 2026 sources describe NVIDIA CUDA passthrough at ~90–100% of native-Linux throughput and within ~5% of native-Windows Ollama, via the driver shim at `/usr/lib/wsl/lib/`. **Critical gotcha:** do *not* install a separate Linux NVIDIA driver inside the guest — that breaks passthrough; only the Windows host driver + the WSL kernel's driver stub should be present. AMD ROCm passthrough is less documented than NVIDIA CUDA. Microsoft previewed a re-architected "WSL3" at Build 2026 (closer-to-native GPU/NPU) but it is preview-only; WSL2 remains the 2026 stable default (W5).
- **nix + TUI inside WSL2 through WT:** works well — both plain Nix (2.34.5 stable, April 2026) inside a Debian/Ubuntu WSL2 guest and NixOS-as-a-WSL-distro (installable as a double-click `.wsl` file on WSL ≥2.4.4) are functional and fast when fronted by WT — confirming ADR108's framing of lane 1 as "the existing flake channel unchanged" (W5).

### 4. The runtime stack on native Windows (lane 2)

- **PostgreSQL + PostGIS:** mature, first-class installer story — official EDB Windows installers plus StackBuilder-distributed PostGIS bundles for PG 14–18 (most recent: PostGIS Windows Bundle 3.6.2, March 16 2026, bundling GEOS 3.14.1/PROJ 8.2.1/SFCGAL 2.2.0/pgSphere). This half of the stack is genuinely solid on native Windows (W6).
- **pgvector — the weak link:** *no* official precompiled Windows binary; the official path is compile-from-source via NMAKE + MSVC (requires a separate Visual Studio Build Tools install). Community precompiled Windows binaries exist (andreiramani/pgvector_pgsql_windows, portalcorp/pgvector_compiled) but are unofficial. A real gap in the "embedded PostgreSQL+PostGIS+pgvector on Windows" story that ADR108's future native-lane packaging ADR must account for (W6).
- **Unix-socket vs TCP-loopback — the nuance ADR108 explicitly flagged, now resolved:** `AF_UNIX` has existed at the Windows OS/API level since Windows 10 build 17063 (stable from 1803), and **PostgreSQL added Unix-domain-socket support on its Windows port starting the PG13 dev cycle** (committed by Peter Eisentraut, 2020-03-28; abstract-namespace sockets added 2020-11-25) — so PG13 and every later version *can* use real Unix sockets on Windows 10 1803+. **However,** `DEFAULT_PGSOCKET_DIR` is empty on the Windows build (there is no Windows-standard `/tmp` analog), so **Postgres defaults to TCP-loopback-only on native Windows unless an admin explicitly sets `unix_socket_directories`.** The accurate statement: Windows *can* do unix sockets for Postgres (not "impossible" as pre-2020 threads claimed), but the zero-config native-Windows behavior is TCP loopback — which **validates ADR108's DSN-resolver TCP-loopback fallback as the correct zero-config native-lane default, not a workaround for an impossibility**, while leaving an explicit unix-socket-dir config open as cheap parity optionality (W6; `ADR108:context`, `ADR108:decision` §3).
- **llama.cpp / Ollama native Windows:** strong — arguably the most mature item in the matrix. Ollama ships a genuine native Windows installer (installs as a Windows service, no WSL/Docker/PATH juggling) with automatic GPU-backend detection for both NVIDIA CUDA and AMD ROCm; llama.cpp ships prebuilt Windows binaries (as of release b9196, ~May 2026) covering CUDA 13.1, Vulkan, HIP, SYCL — caveat: CUDA 13.2 specifically is currently flagged to avoid (12.x–13.1 is the safe range). This confirms `ADR108:context`'s "Ollama/llama.cpp ship native Windows" (W6).
- **Python 3.12:** fully supported, official CPython MSI/EXE installers, Windows 8.1+. The only wrinkle is upstream lifecycle (3.12.13 is late in its bugfix branch; 2026 guidance points newcomers toward 3.14.x) — a general Python-version-clock issue, not a Babylon-Windows-specific gap (W6).

### 5. Rust vs OCaml on Windows — sharpening Part II

- **Rust:** `x86_64-pc-windows-msvc`, `i686-pc-windows-msvc`, and `x86_64-pc-windows-gnu` remain **Tier 1** with host tools. The one 2026-relevant narrowing is `i686-pc-windows-gnu` **demoted from Tier 1 to Tier 2 in Rust 1.88.0 (May 2025)** for low real-world usage — a legacy 32-bit-GNU edge target, not the mainstream 64-bit-MSVC path anyone building Babylon-on-Windows would use. Net: Rust's Windows story is unambiguous, CI-guaranteed, "just works" for the target that matters (W7).
- **OCaml/opam:** opam 2.2 (native-Windows-capable, stable ~2024; 2.5.x current in 2026) is a genuine maturity threshold — it eliminated the old fdopen/Cygwin-shim and gave OCaml a first-class native Windows package manager for the first time, with **two toolchain flavors**: (a) a mingw-w64/GCC port with meaningfully broader package (depext) compatibility, and (b) an MSVC port with *less* external-dependency support, requiring a separate Visual Studio/Build Tools install. A **January-2026** discuss.ocaml.org thread ("Setting up a new OCaml environment in 2026 is not yet a fluid experience") confirms the *live* 2026 gap is no longer the compiler/toolchain (solid) but **package-ecosystem compatibility** — "opam is now native… but not all packages are compatible" with Windows. Net: OCaml crossed a real threshold at opam 2.2 but sits ~1–2 years behind Rust — Rust's remaining Windows gaps are edge-target demotions on a mature platform; OCaml's are still core-package availability on its mainline (mingw) platform, worse on its MSVC variant (W7).

**What AA does to the Part-II verdict — it deepens the OCaml hit.** Part II already recommended against OCaml on Linux-only grounds (hobby-scale interop, two-repo Nix pin, no Aider-Polyglot signal). AA now adds a *binding* axis on which Rust is Tier-1-mature and OCaml is 1–2 years behind with a live, unresolved package-ecosystem gap. Rust's compiled-extension advantage over OCaml therefore **grows** under the Windows horizon, while OCaml's single genuine win (GADT exhaustiveness/functor packaging, Part II §1) is platform-independent and entirely unchanged by AA. The Part-II verdict — *keep the algebra in Python; if it must be compiled, extend the existing Rust/PyO3 crate* — is reinforced, not challenged, and the exhaustiveness discipline is still best recovered as Python predicate specs + property tests per your Fable-5 doctrine (W7; Part II §2–3) **[judgment]**.

### Bottom line for Part III

Amendment AA moves **no verdict** in Parts I–II; it hardens the ground under each. Textual and the Neovim hybrid both port — WSL2 verbatim, native-Windows with the glyph floor carrying the raster gap (Textual's Sixel-only / no-Kitty and Neovim's image.nvim-`ioctl` break degrade to the *same* glyph floor). Emacs weakens further (WSL2 easy, native adds AV/MSYS2 friction and has no native terminal graphics even in trunk, GUI-vs-TTY parity under-verified). OCaml's disadvantage deepens (Rust Tier-1 on Windows vs OCaml's still-open package-ecosystem gap). The two seams ADR108 pre-named as carrying the optionality both check out under verification: the DSN-resolver TCP-loopback fallback is the *correct* zero-config native-Windows default (not a hack), and the ADR099 glyph floor is precisely the rendering layer that survives both Windows lanes. Every recommendation stands **[judgment]**.

## Appendix A — Costs and Risks

**Textual / v1.0 path**
- *Bus-factor:* single-maintainer post-Textualize (S5 §1). *Mitigation:* the seam is text/markdown, not Textual internals (S4 §3); a year of healthy company-free releases already de-risks it (S5 §1); disposability ruling keeps a rebuild cheap. *(Tension noted honestly: this "healthy sustained cadence" reading and the "five majors in 12 months = churn tax" reading below describe the **same** release history — reassuring on survival, taxing on upgrades.)*
- *Version churn:* five majors in 12 months; pins are not stability signals (S5 §1). *Mitigation:* Nix pin (`textual>=8.2,<8.3`) + read-changelog-before-bump discipline (S4 §2, S5 §1).
- *Perf ceilings:* DataTable + GIL/asyncio (S5 §1). *Mitigation:* turn-based simulation is unlikely to hit them **[judgment]**; `textual-fastdatatable` exists if it does.
- *ADR-vs-code drift:* ADR099 (lines 49–50) says "SIXEL IS NOT A TARGET" yet `capability.py:126–139` still probes sixel as a fallback — a minor documented-nowhere discrepancy, not a bug; one of the few externally-checkable in-tree claims that holds exactly (S4 §2). *Action:* reconcile or note it.
- *Snapshot re-bake:* ArchiveApp-layout changes re-bake all SVG goldens (MEMORY) — but SVG snapshots are aesthetic-tier only per T6, never a behavioral gate (S4 §4), so this is noise, not risk.
- *Live loop is unbuilt:* the paced driver does not exist yet; `AsyncSimulationRunner` is dead-on-arrival (zero callers, wrong facade); T4 builds a fresh Textual-worker driver against the `advance_tick`/`PausePredicate` seam (S4 §6). This is the main remaining *work* (and the reason the platform's "no new platform work" status must not be read as "v1.0 is nearly free"), not a platform risk.

**Neovim / post-1.0 second client**
- nvim-treesitter archived April 2026 → pin built-in 0.12+ treesitter, budget parser install in the closure (S2 §1). pynvim thin (0.6.0, 33 issues) → expect to file/fix edge cases (S2 §2). No published ext_ui latency → benchmark, but a local-socket hop is ~sub-ms and likely imperceptible for turn-based, so due diligence not blocker (S2 §2). `vim.ui.img` is 0.13-nightly-only → images via image.nvim/snacks or a vendored pinned nightly (S2 §3,§4). GNOME-Terminal image gap → glyph floor mandatory regardless (S2 §4). Modal barrier for casual players → game keymap layer or narrowed audience (S2 §6). Second language ecosystem in the closure — *size delta trivial vs the 4.49 GB baseline; the real cost is the standing second pin/cache/version story* (S4 §4). KSBC = new surface incl. hand-rolled 256-color degrade (S4 §5). Client-side re-implementation of the verb write-path + verb plates + pause UI (not in vault markdown) (S4 §3,§6).
- **Standing maintenance surface (the decisive solo-dev cost):** a Neovim second client means one person maintaining Python + Lua + pynvim edge-cases + a parallel `mini.test` harness **indefinitely** post-1.0. "Disposability sanctions it" addresses *build* cost, not this ongoing *two-client* tax — tally it before committing (S2 §2,§5; S4 §4).

**Emacs (not recommended)**
- First-of-kind game UI (no transient/buffer game precedent, S3 §1). TTY child frames are Emacs-31-*trunk*, not a shipped-stable strength (current release 30.1, S3 §2). native-comp↔Nix immutable-store friction (S3 §4). No native terminal graphics even in 2026 trunk (S3 §2). Casual-hostile baseline vocabulary (S3 §6). Heaviest maintenance tail (S3 §4, S5 §4). Daemon commitment to get acceptable startup (S3 §4). Elisp essentially absent from AI-codegen leaderboards (§6).

**OCaml (not recommended for the algebra layer)**
- Hobby-scale Python interop (pyml ~205 stars vs PyO3 industrial, S1 §4). Two-repo Nix pin (S1 §2). Absent from *Aider Polyglot specifically* (not absence of all signal; MultiPL-E may cover it, S1 §7). Third language in a pinned closure. Stale flagship docs (RWO 2022 vs OCaml 5.5, S1 §7). Onboarding friction ("not yet a fluid experience," S1 §1). *Genuine loss if not adopted:* native GADT exhaustiveness + functor packaging (S1 §6) — recovered as Python predicate specs + property/exhaustiveness tests per Fable-5 doctrine **[judgment]**. *Reversible:* the algebra layer doesn't exist yet, so "not now" can become "later" at no option cost.

**Cross-cutting**
- Float/libm transcendental non-reproducibility is identical across Python/OCaml/Rust and already governed by your tolerance-policy discipline (S1 §5) — not a differentiator.
