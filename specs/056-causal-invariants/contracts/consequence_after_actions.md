# Contract: Consequence After All OODA Actions (US2, INV-014)

**Invariant ID**: INV-014
**User Story**: US2 (P1) — Consequence Systems run after all OODA actions
**Constitutional Basis**: Constitution I.17 OODA; III.7 Determinism (replay requires order-independence over the organization set)
**Test File**: `tests/property/invariants/test_consequence_after_actions.py`

**Pre-test prerequisite (F6=α, 2026-05-07)**: This test depends on the
Phase 2 reorder of `_DEFAULT_SYSTEMS` so `OODASystem` runs at
position 14 (between `MetabolismSystem` and `SurvivalSystem`). Without
the reorder, every Consequence System trivially fails the predicate
(they all run *before* OODA in current order, never after). The
reorder is part of T004's production work; the F1 helper extraction
(`OODASystem._resolve_for_organization`) is the spy-injection seam
T011 patches.

## Predicate

```text
For every full SimulationEngine.run_tick() invocation that includes OODASystem:
  For every Consequence System C in CONSEQUENCE_SYSTEMS that was invoked this tick:
    monotonic_timestamp_ns(C.step) > max(
      monotonic_timestamp_ns(action) for action in OrganizationActionSpy.events
    )

AND, separately:

For every starting WorldState with >= 2 organizations:
  Running run_tick twice with the same seed but with the per-organization
  iteration order shuffled inside OODASystem produces post-tick states
  equal under Spec 055 exclude rules (Acceptance Scenario 3 — order
  independence over the organization set).
```

In plain English: no Consequence System may execute until **every
organization** has resolved its OODA action for the current tick.
Initiative order may not silently determine outcomes downstream.

## Inputs

- `SimulationEngine` constructed with the default System list
- A random `WorldState` generated by `worldstate_strategy(min_entities=2)`
  (so at least 2 organizations exist for the per-org spy to record)
- A `ServiceContainer.testing()` instance
- A `TickContext` with `tick=N` for an arbitrary N

## Outputs

- A `TickTrace` containing both `system_calls` and `org_actions`
- For each Consequence System invoked this tick: its
  `monotonic_timestamp_ns` is strictly greater than the latest
  `OrganizationActionEvent.monotonic_timestamp_ns`

## Falsification

| Condition | Failure Mode |
|-----------|--------------|
| Any Consequence System's call timestamp is ≤ any organization's action timestamp | Interleaving detected; failure message names `(consequence_system, organization_id)` pair where the violation was first observed |
| Acceptance Scenario 3: two runs with shuffled per-org iteration order produce different post-states | Order-dependence detected; failure message shows the diff of the two post-state `model_dump`s |
| `CONSEQUENCE_SYSTEMS` does not classify any of the systems Spec assumes are consequences | Import-time `AssertionError` from `simulation_engine.py` partition check; test never runs |

## Implementation Notes

- `OrganizationActionSpy` wraps the per-organization loop iteration inside
  `OODASystem.step` (`ooda.py:79`) via `unittest.mock.patch.object` (per
  research.md §2). The spy harness's docstring records the line-number
  assumption.
- `SystemCallSpy` provides the consequence-System call timestamps; the
  spies share a `monotonic_timestamp_ns` clock domain (`time.monotonic_ns()`).
- For Acceptance Scenario 3, the test runs `engine.run_tick` twice on
  deep-copies of the same starting `WorldState`, with the second run
  patching the per-org iteration order via a deterministic shuffle. Both
  runs use the same Hypothesis-derived seed so any non-determinism unrelated
  to org-order is excluded as a confounder.
- The "shuffled order" patch is implemented by sorting the
  `org_nodes[:max_orgs]` list with a deterministic but reversed key, so
  the two runs see organizations in opposite orders.
- Empty action phases (zero organizations) trivially satisfy the predicate
  — the test records "0 organizations, 0 interleavings" rather than
  skipping.
