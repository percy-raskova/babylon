# Contract: No DB I/O During Tick (US3, INV-015)

**Invariant ID**: INV-015
**User Story**: US3 (P2) — No DB I/O during tick execution
**Constitutional Basis**: Constitution II.6 ("No DB I/O during tick" — verbatim); II.10 World Runtime; II.11 Subsystem Table Ownership; ADR037 Postgres Runtime
**Test File**: `tests/property/invariants/test_no_db_io_during_tick.py`

## Predicate

```text
For every random WorldState generated by worldstate_strategy():
  with no_db_io_during_tick(services):
    engine.run_tick(graph, services, context)

Must not raise DBIONotPermittedError.
```

In plain English: from the moment `engine.run_tick(...)` is called until
it returns, no System may access any DB-bearing service exposed by
`ServiceContainer`. Hydration (before `run_tick`) and persistence (after
`run_tick`) ARE permitted to touch the database — they happen outside the
patched scope (per clarification Q3).

## Inputs

- `SimulationEngine` constructed with the default System list
- A random `WorldState` generated by `worldstate_strategy()` from Spec 040
- A `ServiceContainer.testing()` instance with hydration already complete
- A `TickContext` with `tick=N` for an arbitrary N

## Outputs

- `engine.run_tick(graph, services, context)` returns successfully
- No `DBIONotPermittedError` is raised inside the `no_db_io_during_tick`
  scope

## Falsification

| Condition | Failure Mode |
|-----------|--------------|
| Any System touches a patched DB surface during `run_tick` | `DBIONotPermittedError` raised by the sentinel; failure message names the System whose `step()` was running and the DB surface accessed (`services.database.execute`, `services.persistence.persist_tick`, etc.) |
| Acceptance Scenario 3: hydration before the patch or persistence after the patch is incorrectly blocked | Test failure outside the patched scope — indicates the patch boundary is incorrectly placed |
| A new DB-bearing service is added to `ServiceContainer` without being detected by the introspection | A System touching the new service does not raise — the introspection regex (research.md §4) needs an update |

## Implementation Notes

- `no_db_io_during_tick(services)` introspects
  `dataclasses.fields(services)` and patches every field whose declared
  type or attribute name matches the regex
  `(?i).*(database|persistence|runtime|store).*` (research.md §4).
- Each patched field is replaced with a sentinel object whose
  `__getattr__` raises `DBIONotPermittedError(field_name, attr_name)`.
- Patch entry: at the `with` statement.
- Patch exit: at `with` block exit (including exception path) — originals
  are restored from a saved snapshot.
- Acceptance Scenario 3 is exercised by a separate test that runs
  hydration *before* entering the patched scope and persistence *after*
  exiting — both succeed, demonstrating the patch boundary precisely
  matches `run_tick`.
- Surface inventory at time of writing (research.md §4):
  - `database: DatabaseConnection` → patched
  - `persistence: Any` → patched if non-None and matches regex
  - `metrics: MetricsCollector` → NOT patched (in-memory)
  - `event_bus: EventBus` → NOT patched (in-memory pub/sub)
  - `formula_registry: FormulaRegistry` → NOT patched (pure functions)
  - `config: SimulationConfig` → NOT patched (frozen Pydantic model)

## Default-Deny + Opt-Out Marker

A System that legitimately requires read-only DB access during tick
execution (none currently exist; this is exactly the constitutional
prohibition US3 enforces) MUST flag itself with:

```python
class SomeSystem:
    bypasses_causal_invariant: ClassVar[dict[str, str]] = {
        "no_db_io_during_tick": (
            "Reads <static reference table> that cannot be hydrated up "
            "front because <reason>. Read-only; no writes."
        ),
    }
```

The harness consumes this marker, narrows the patch to exclude that
System's reads of the named table, and machine-asserts that the
justification is non-empty (FR-011). **Adding such a marker is a
constitutional change** — a maintainer who adds one is bypassing II.6
and should escalate per Constitution IX governance rules.
