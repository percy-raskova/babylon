# Feature Specification: Vol II Circulation System with LODES OD Integration

**Feature Branch**: `063-vol-ii-circulation`
**Created**: 2026-05-13
**Status**: Draft
**Input**: User description: "One spec (063: Vol II + LODES) — T054, T055, T079, T080. Carry forward the four spec-062 tasks deferred as 'downstream spec scope': LODES origin-destination matrix loader, the Vol II circulation step that consumes it, the wiring that closes Φ inflow from external nodes into US counties through the existing ImperialRentSystem tick, and the Canada-bound commute special-case mandated by Constitution IV.1."

## Overview

This feature lands the **Vol II Circulation** stage of the five-flow per-tick pipeline established by spec 062, plus the two boundary-routing seams that close Constitution IV.1 (Detroit-Windsor) for the production engine.

Spec 062 ratified five flow stages — Production → Imperial Rent inflow → Circulation → Equalization → Distribution — and shipped four of them. **Circulation was deferred** because it depends on the LODES (LEHD Origin-Destination Employment Statistics) commute matrix, which Constitution II.13 names as the "min-cost flow" component of the Transport Substrate. The deterministic LODES component is well-specified federal data; the emergent slime-mold conductivity component of II.13 is **explicitly out of scope** for this feature and remains deferred to a future spec.

In addition, two seams remain open from spec 062:

1. The Φ-distribution helper `distribute_phi_week_to_counties()` exists as a sibling module but is **not yet called** from the per-tick `ImperialRentSystem.step()`. Wiring this closes the FR-034/FR-035 contract for the live engine and makes Constitution IV.1's Canada → US-county Φ flow operational rather than just helper-callable.
2. The Detroit-Windsor commute corridor needs its own routing rule: when LODES routes a worker from a tri-county hex to a Windsor (Ontario) destination, the boundary register row must record `dest_node_id='canada'` rather than the Rest-of-USA fallback. Spec 062 added Canada as a first-class external node (FR-036, ratified in research §4) precisely so this rule has a concrete destination to point at.

This feature is the integration layer that turns four already-built primitives into a fifth running flow stage and closes the two related boundary seams.

## Clarifications

### Session 2026-05-13

- Q: Which LODES job-type segment should the OD matrix loader consume — JT00 (all jobs), JT01 (primary jobs), JT02 (private-sector), or JT03 (federal)? → A: **JT00 (all jobs)** — Babylon is a simulation for a game; every employment relationship is a meaningful thread of economic engagement, so the OD matrix captures the full labor-allocation surface rather than the academic primary-job abstraction. Multi-job workers contribute to multiple OD rows; conservation arithmetic operates on the resulting weighted-share form (FR-009) which is well-defined regardless of double-counting at the worker level.
- Q: Should the LODES OD loader load the all-workers aggregate (S000) only, or stratify by worker segment (age SA*, earnings SE*, industry SI*)? → A: **S000 only** (all-workers aggregate) — keeps the spec coherent with FR-013's hex-aggregated `v` and spec 062 FR-031's derive-on-read industry pattern. Stratification by earnings or industry is deferred to a future spec when downstream consumers (consciousness layer, class-position dynamics, Vol III equalization at sub-NAICS granularity) actually exist. Loading 3× or 9× larger matrices now without a consumer is premature optimization.
- Q: For cross-border commute (Detroit→Windsor or any out-of-area destination), should the spec record only the variable-capital outflow (`COMMUTE_OUT`) or also pair it with a wage-repatriation inflow row? → A: **Pair `COMMUTE_OUT` with a `TRADE_EDGE` inbound row** representing the worker's wage flowing back to the origin US household. The paired emission applies to ALL boundary-exiting commute (not just Canada): Detroit→Windsor and Detroit→Toledo both produce two rows per OD entry. The paired `TRADE_EDGE` row is **observational only** — it does NOT re-add to the in-study-area `v` vector during Circulation, so spec 062's per-tick conservation invariant FR-010 remains unchanged. The pairing exists so a future Department III household-reproduction spec can attribute the repatriated wage flow without back-fitting the boundary register schema.
- Q: Web research (2026-05-13) confirmed that no public dataset publishes Detroit-Windsor commute as LODES-equivalent OD microdata. Should the spec defer real Canada-bound row population entirely to a future spec, or bolt on a small **BTS + WWE synthesis loader** that produces aggregate weekly Canadian-bound rows? → A: **Bolt on a synthesis loader** (Option B). Add a `BorderCommuteSynthesisLoader` that reads (a) BTS Border Crossing Data monthly CSV for Ambassador Bridge + Detroit-Windsor Tunnel personal-vehicle southbound counts and (b) StatCan Frontier Counts monthly CSV for the Canada-bound northbound counterpart, applies a configurable `border_commute_share` anchor (default 0.50, traced to Workforce WindsorEssex 2017 report's ~6,120 / ~12K daily-vehicles ratio), and produces one CMA-to-tri-county aggregate Canadian-bound flow per simulated week. These synthetic rows merge into the loaded LODES OD matrix at session init time as additional sparse entries with `workplace_dest='canada'` so the existing classifier and circulation step handle them uniformly. The synthesis is gated behind a config flag (`enable_border_commute_synthesis`, default `False` for back-compat); when disabled, behavior is identical to the LODES-only path.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Vol II Circulation Runs Per-Tick with Conservation (Priority: P1)

A simulation engineer runs a Detroit tri-county session for one tick. After the Production stage has assigned variable capital `v` to each producing hex, the Vol II Circulation stage redistributes that variable capital across hexes according to the empirically-grounded LODES commute matrix — workers commute, so the variable capital they represent flows to the hex where they actually work. Within the study-area boundary, the total `v` is conserved exactly; for commute flows that route a worker to a destination outside the tri-county study area, the variable capital exits via the Boundary Flow Register so per-tick conservation balances at the boundary.

**Why this priority**: This is the load-bearing economic mechanic that makes the five-flow pipeline functionally complete. Without Circulation, the simulation conflates where value is *produced* with where it is *realized* — a category error in Marxist political economy that breaks downstream Vol III equalization (which compares profit rates across the *real* labor distribution, not the production-side phantom). P1 because the entire Vol II surface of the simulation depends on it.

**Independent Test**: Construct a contrived two-hex Detroit scenario (hex A in Wayne County, hex B in Oakland County). Seed Production output so each hex carries `v = $1,000,000`. Synthesize a LODES OD matrix routing 30% of B's workers to A and 5% of A's workers to a Windsor (Canada-side) destination outside the study area. Step the Circulation system once. Verify (a) `v[A] = 1,000,000 + 0.3 × 1,000,000 = 1,300,000`, (b) `v[B] = 0.7 × 1,000,000 = 700,000`, (c) the Boundary Flow Register has one row with `flow_type=COMMUTE_OUT` and `magnitude = 50,000` (representing the 5% of A's workers exiting), (d) sum of in-study-area `v` plus the boundary register exit row equals exactly the pre-circulation total — no value created or destroyed.

**Acceptance Scenarios**:

1. **Given** a tri-county session at tick t with `v` seeded by Production, **When** the Circulation stage runs, **Then** the post-circulation sum of `v` over all in-study-area hexes plus the magnitude of all `COMMUTE_OUT` boundary register rows added that tick equals the pre-circulation `v` sum within floating-point tolerance.
2. **Given** a LODES OD matrix where row j sums to S_j > 0 (worker presence at origin), **When** Circulation runs, **Then** the share of `v[j]` routed to destination A equals `OD[j, A] / S_j`, never absolute `OD[j, A]` directly.
3. **Given** a hex j whose LODES row sum is zero (no observed workers — uninhabited cell), **When** Circulation runs, **Then** the system MUST NOT divide by zero; `v[j]` carries forward unchanged and no boundary register rows are emitted from j.
4. **Given** the same OD matrix and seed used twice in the same session, **When** Circulation runs at tick t and tick t+1 against identical pre-state, **Then** the post-state and emitted boundary rows are bit-identical (deterministic — the matrix is a year-scoped constant per Constitution II.13's min-cost flow form).
5. **Given** the simulation engine runs without LODES data hydrated for the active simulated year, **When** initialization completes, **Then** the engine MUST refuse to start with an explicit error naming the missing year, rather than silently routing all `v` as if no commute occurred.

### User Story 2 - Φ Inflow Reaches US Counties Via ImperialRentSystem (Priority: P1)

A simulation engineer runs a Detroit session for one tick. Hickel's drain time series declares an annual Φ_year inflow into the United States from the international periphery. The Imperial Rent stage of the per-tick pipeline must distribute one week's worth of that Φ to the actual US counties in the study area, weighted by their import-share-weighted exposure to the periphery — not deposit it into a generic Rest-of-USA node, not loop back to the source external nodes. The wiring must use the existing `distribute_phi_week_to_counties()` helper as the engine's per-tick action, so that what was previously a callable utility becomes the live engine behavior.

**Why this priority**: This closes a known seam from spec 062 — the helper exists and is unit-tested, but the engine system hasn't been switched over to use it. Until the wiring lands, `ImperialRentSystem.step()` does not actually distribute Φ to counties during a real tick, meaning Constitution IV.1's claim that "Canada-specific Φ flows into US counties via the standard drain-edge mechanism" is theoretical rather than operational. P1 because the helper-vs-engine gap is a silent correctness bug: tests pass on the helper, but a live tick produces no county-level Φ inflow rows.

**Independent Test**: Initialize a Detroit session with a Hickel drain reference value of `Φ_year = $52,000,000` from Canada for the simulated year (so `Φ_week = $1,000,000`). Provide a county-exposure weight vector that gives Wayne County 0.5, Oakland County 0.3, Macomb County 0.2 of the import-share-weighted exposure. Run `engine.run_tick()` for one tick. Query the boundary flow register for `flow_type=DRAIN_EDGE` rows where `source_node_id='canada'`. Verify three rows: Wayne `magnitude=500,000`, Oakland `300,000`, Macomb `200,000`. Sum equals `Φ_week` exactly within tolerance, and no row points at `dest_node_id='rest_of_usa'`.

**Acceptance Scenarios**:

1. **Given** an external node carrying `phi_year_inflow > 0` for the active simulated year, **When** `ImperialRentSystem.step()` runs at tick t, **Then** the boundary flow register contains exactly one row per (external source, county destination) pair with `flow_type=DRAIN_EDGE`, `source_kind=external`, `dest_kind=county`, and the sum of magnitudes from that source equals `phi_year_inflow / 52` within tolerance.
2. **Given** a session where county exposure weights sum to 1.0 across all internal counties, **When** the Φ distribution runs for any external source, **Then** no Φ magnitude is recorded against `dest_node_id='rest_of_usa'` (Φ targets specific US counties, not the residual node).
3. **Given** an external node with `phi_year_inflow = 0` (no drain that year — pre-1995 or unmapped partner), **When** `ImperialRentSystem.step()` runs, **Then** zero `DRAIN_EDGE` rows are emitted from that source for that tick.
4. **Given** the same session run for 52 consecutive ticks within a single simulated year, **When** boundary register rows for `flow_type=DRAIN_EDGE` are summed by source, **Then** each external source's annual total equals its declared `phi_year_inflow` within a residual no greater than 52 × ε (one floating-point unit per weekly distribution).

### User Story 3 - Detroit-Windsor Commute Routes to Canada (Priority: P2)

A simulation engineer runs a Detroit tri-county session. The LODES Detroit OD matrix includes commuter flows from US tri-county hexes to workplace destinations in Windsor, Ontario (the cross-border counterpart). When the Circulation stage processes one of these cross-border flows, the resulting Boundary Flow Register row must declare `dest_node_id='canada'` and `dest_kind=external` — not the generic Rest-of-USA fallback that captures domestic out-of-area commute. This makes the Detroit-Windsor corridor an empirically-grounded boundary surface (Constitution IV.1) rather than a categorically-collapsed approximation.

**Why this priority**: Constitution IV.1 is non-negotiable, but the engine-level routing rule that distinguishes Canadian from domestic out-of-area commute is the only piece still missing. Once Vol II circulation (US1) is operational, this is a single classification rule applied per emitted boundary row. P2 because (a) it depends on US1 landing first, (b) without it US1 still works correctly for in-study-area conservation, just with a less accurate dest_node attribution for the small subset of cross-border flows.

**Independent Test**: Set up a contrived OD row from a Detroit hex routing 100 workers to a Windsor destination. The Windsor destination's FIPS-equivalent code falls outside Michigan/Ohio/Indiana/Illinois. Run Circulation for one tick. Verify exactly one boundary register row with `source_kind=hex`, `source_node_id=<the Detroit hex H3 index>`, `dest_kind=external`, `dest_node_id='canada'`, `flow_type=COMMUTE_OUT`, magnitude equal to the variable capital share corresponding to those 100 workers.

**Acceptance Scenarios**:

1. **Given** a LODES OD row routing workers from a US tri-county hex to a Windsor destination, **When** Circulation runs, **Then** the emitted boundary register row carries `dest_node_id='canada'`, never `'rest_of_usa'`.
2. **Given** a LODES OD row routing workers from a US tri-county hex to Toledo (Lucas County, Ohio — out of study area but domestic), **When** Circulation runs, **Then** the emitted row carries `dest_node_id='rest_of_usa'`, not `'canada'`.
3. **Given** a Windsor-bound flow and a Toledo-bound flow in the same tick from the same origin hex, **When** Circulation runs, **Then** two distinct boundary register rows are emitted with the correct destination attribution for each.
4. **Given** the simulation runs without Canada present in the external-node registry (regression check on Constitution IV.1), **When** the routing rule fires for a Windsor destination, **Then** initialization MUST have refused to start in the first place — the rule never reaches a session where the destination node is unresolvable.

### Edge Cases

- **LODES OD row with only out-of-study-area destinations**: All variable capital from that origin hex flows out the boundary. Conservation must hold: the origin hex's `v` post-circulation goes to zero, and the boundary register absorbs the entire pre-circulation `v` for that origin via one or more `COMMUTE_OUT` rows summing exactly to the original `v[origin]`.
- **OD destination outside both LODES coverage and the study area**: For destinations LODES does not enumerate (data gap rather than known out-of-area), the system MUST NOT silently drop the flow. It MUST route to the most-specific available boundary node (`rest_of_usa` if no better match) and emit an audit log entry naming the unmapped destination.
- **Φ distribution with all-zero exposure weights**: A degenerate session where no county has been assigned any import-share exposure. The Φ helper MUST refuse to distribute (not silently divide by zero); the engine MUST surface this as a configuration error during the first tick rather than silently producing zero `DRAIN_EDGE` rows.
- **LODES year and simulated year mismatch**: LODES is published for specific calendar years. If the simulated year falls outside LODES coverage, the engine uses the nearest covered year's matrix (mirroring FR-041's clamp behavior for Hickel) and emits one warning per session rather than per tick.
- **Hex pair appears in LODES but not in the H3 grid loaded by the session**: A LODES origin or destination block-group references a hex H3 cell the active study area does not contain. The system MUST drop that cell's contribution from the in-study-area loop and route it to `rest_of_usa` rather than fabricating membership.
- **Determinism under matrix mutation**: The LODES OD matrix MUST be treated as immutable for the duration of a simulated year. If a downstream system needs to apply year-rollover (e.g., switch from the 2010 matrix to the 2011 matrix at the year boundary), that rollover MUST happen at a tick boundary and MUST be recorded in the audit log, never mid-tick.
- **Cross-border `COMMUTE_OUT` without paired `TRADE_EDGE`**: A bug or partial implementation emits the outflow row but skips the wage-repatriation pairing. The conservation auditor MUST detect the missing pair (FR-030c) and surface an alarm in the same tick, not silently leak the round-trip information.
- **Paired `TRADE_EDGE` magnitude is observational only**: Downstream consumers MUST treat the inbound `TRADE_EDGE` from FR-030a as an annotation, not as a v-vector update. Naively summing it back into the in-study-area `v` would double-count the wage flow and break FR-010 conservation.

## Requirements *(mandatory)*

### Functional Requirements

**LODES OD Matrix Loader (FR-001 to FR-007)**

- **FR-001**: The system MUST load the LODES Origin-Destination commute matrix from the existing on-disk LODES dataset (currently at `data-trove/lodes/od/`) into a runtime sparse matrix representation conforming to Constitution II.12's matrix layer requirements.
- **FR-001a** *(added 2026-05-13 via Clarifications)*: The loader MUST consume the **JT00 (all jobs)** LODES job-type segment, not JT01/JT02/JT03. Every employment relationship a worker holds contributes to the OD matrix; multi-job workers may appear in multiple origin-destination rows. This captures the full labor-allocation surface for the game simulation rather than the academic primary-job abstraction. Conservation arithmetic (FR-009, FR-010) operates on the row-sum-normalized weighted-share form, which is well-defined regardless of multi-job double-counting at the worker level.
- **FR-001b** *(added 2026-05-13 via Clarifications)*: The loader MUST consume the **S000 (all-workers aggregate)** segment only. Worker-segment stratification (LODES SA*/SE*/SI* dimensions for age, earnings, and broad industry) MUST NOT be loaded as separate matrices in this spec. Stratification is deferred to a future spec when downstream consumers (consciousness layer, class-position dynamics, sub-NAICS Vol III equalization) require the additional dimensions. This keeps the loaded matrix coherent with FR-013's hex-aggregated `v` contract and spec 062's FR-031 derive-on-read industry pattern.
- **FR-002**: The loaded matrix MUST carry origin and destination identifiers at the H3 res-7 hex resolution used elsewhere in the simulation, derived from the LODES block-group-to-hex crosswalk (the existing `us_xwalk.csv.gz` fixture).
- **FR-003**: The matrix MUST be year-scoped: one matrix per simulated year, loaded at session initialization for every year the scenario spans, mirroring spec 062's reference-data hydration contract.
- **FR-004**: For LODES years that are missing from the on-disk dataset for the active study area, the system MUST clamp to the nearest available year (FR-041 in spec 062's clamp semantics), emit one audit log entry per session naming the substitution, and proceed.
- **FR-005**: The loader MUST be deterministic: loading the same LODES year twice in two distinct sessions MUST produce matrices whose nonzero entries are bit-identical.
- **FR-006**: The matrix MUST be treated as immutable within a simulated year; no system MUST mutate it after load.
- **FR-007**: The matrix MUST scope its rows and columns to the active study area's hex set plus the destinations that any in-study-area row routes to (rows that route to no in-study-area destination MUST be dropped to keep memory bounded; their workers exit via `COMMUTE_OUT` rows in the Circulation stage).

**Vol II Circulation Step (FR-008 to FR-016)**

- **FR-008**: The per-tick pipeline MUST execute a Circulation stage between the Imperial Rent inflow stage and the Equalization stage, satisfying spec 062's FR-025 ordering contract.
- **FR-009**: The Circulation step MUST move variable capital `v` between hexes per the formula `v[A, t+1] = sum_j(OD[j, A] × v[j, t] / row_sum[j])` for every in-study-area destination A, where `row_sum[j]` is the total LODES outflow from origin j.
- **FR-010**: The Circulation step MUST conserve the total in-study-area `v` plus boundary-exit `v` exactly within floating-point tolerance: `sum(v post-circulation) + sum(COMMUTE_OUT magnitudes this tick) = sum(v pre-circulation)`.
- **FR-011**: When `row_sum[j] = 0` for an origin j, the Circulation step MUST carry `v[j]` forward unchanged and emit no boundary rows from j (no division by zero).
- **FR-012**: For every OD entry routing variable capital from an in-study-area origin to an out-of-study-area destination, the Circulation step MUST emit one `COMMUTE_OUT` row in the Boundary Flow Register with the correct `source_node_id` (the origin H3 cell), `source_kind=hex`, `dest_kind` per the destination's classification, and `magnitude` equal to that share of the origin's `v`.
- **FR-013**: The Circulation step MUST NOT mix variable capital across industries — it operates on hex-aggregated `v`, mirroring spec 062's FR-031 industry-derivation-on-read pattern. Per-industry circulation is out of scope for this spec.
- **FR-014**: The Circulation step MUST be deterministic: identical pre-state plus identical OD matrix MUST yield bit-identical post-state and boundary register rows.
- **FR-015**: The Circulation step MUST execute as the third sub-stage within the existing `ImperialRent` slot of the per-tick pipeline (spec 062 FR-053's 5a–5d sub-stage decomposition: 5a Production, 5b Imperial Rent inflow, 5c Circulation, 5d Equalization, 5e Distribution), preserving spec 062's slot contract.
- **FR-016**: The Circulation step MUST be representable as a single sparse matrix-vector multiplication for the in-study-area portion (Constitution II.12), with the out-of-area `COMMUTE_OUT` rows derivable from the row-sum residual.

**Φ Distribution Engine Wiring (FR-017 to FR-022)**

- **FR-017**: `ImperialRentSystem.step()` MUST invoke the existing `distribute_phi_week_to_counties()` helper exactly once per tick, supplying it the active session's external nodes, the per-county import-share-weighted exposure weights, and the runtime Boundary Flow Register.
- **FR-018**: For each external node with `phi_year_inflow > 0` for the active simulated year, the engine MUST distribute `phi_year_inflow / 52` across counties weighted by exposure, recording one `DRAIN_EDGE` row per (external source, county dest) pair.
- **FR-019**: Φ distribution rows MUST land at `dest_kind=county` — never `dest_kind=external` (which would loop Φ back to a sibling external node) and never `dest_kind=hex` (which would over-attribute precision Hickel's annual data does not support).
- **FR-020**: For external nodes with `phi_year_inflow = 0` for the active year (pre-1995 unmapped, or partner countries with no Hickel coverage), the engine MUST emit zero `DRAIN_EDGE` rows from that source for that tick, but MUST NOT fail the tick.
- **FR-021**: The annual sum of `DRAIN_EDGE` magnitudes from each external source over 52 consecutive ticks within a single simulated year MUST equal `phi_year_inflow` within a residual of 52 × ε.
- **FR-022**: The wiring MUST preserve spec 062's FR-008a per-tick transactional atomicity: all `DRAIN_EDGE` rows emitted within a single tick land in the same per-tick transaction envelope and commit atomically with all other tick state changes.

**Detroit-Windsor Commute Routing (FR-023 to FR-029)**

- **FR-023**: When the Circulation step emits a `COMMUTE_OUT` row for a destination whose LODES classification identifies it as a Canadian (Ontario) workplace area, the row MUST carry `dest_node_id='canada'` and `dest_kind=external`.
- **FR-024**: When the Circulation step emits a `COMMUTE_OUT` row for a domestic destination outside the tri-county study area (e.g., other Michigan counties, Ohio, Indiana, Illinois), the row MUST carry `dest_node_id='rest_of_usa'` and `dest_kind=external`.
- **FR-025**: The Canadian-vs-domestic classification rule MUST use the LODES destination identifier's encoded country/state code, never a string comparison on a free-text destination name.
- **FR-026**: Initialization MUST refuse to start if the routing rule could fire (i.e., the LODES matrix contains any Windsor-side destinations) but the external-node registry does not contain `canada` (Constitution IV.1 invariant — Canada must exist before any flow can route to it).
- **FR-027**: The classification rule MUST be applied row-by-row at emission time, not as a post-processing pass over already-emitted rows, so that the audit log captures the correct destination from the first commit.
- **FR-028**: For LODES destinations outside both the study area AND known categorical buckets (e.g., a workplace block-group in a US territory not enumerated in the FIPS state-code set), the row MUST default to `dest_node_id='rest_of_usa'` and emit one audit log entry per unique unmapped destination per session.
- **FR-029**: The conservation invariant from FR-010 MUST hold across both Canadian and domestic boundary routing — splitting an origin's outflow between `'canada'` and `'rest_of_usa'` rows MUST sum exactly to the origin's pre-circulation outflow share, with no double-counting and no leakage.
- **FR-030a** *(added 2026-05-13 via Clarifications)*: For every `COMMUTE_OUT` row emitted with `dest_kind=external` (i.e., crossing the study-area boundary to either `canada` or `rest_of_usa`), the system MUST emit a paired `TRADE_EDGE` row in the same tick representing wage repatriation back to the origin US hex. The paired row carries `source_node_id=<the external destination of the COMMUTE_OUT>`, `source_kind=external`, `dest_node_id=<the origin hex of the COMMUTE_OUT>`, `dest_kind=hex`, `flow_type=TRADE_EDGE`, and `magnitude=<the COMMUTE_OUT magnitude>` (full wage repatriation assumed in v1; refinement for taxes / on-site consumption deferred to a future Department III household-reproduction spec).
- **FR-030b** *(added 2026-05-13 via Clarifications)*: The paired `TRADE_EDGE` row from FR-030a is **observational only** — it MUST NOT modify the in-study-area `v` vector during the Circulation stage, and the conservation invariant FR-010 MUST remain stated as `sum(v post-circulation) + sum(COMMUTE_OUT magnitudes) = sum(v pre-circulation)` (the inbound `TRADE_EDGE` does not re-enter the v-conservation arithmetic). This preserves spec 062's per-tick conservation contract while making the round-trip auditable for future household-reproduction analysis.
- **FR-030c** *(added 2026-05-13 via Clarifications)*: The pairing rule from FR-030a MUST be applied at emission time — a `COMMUTE_OUT` with `dest_kind=external` written to the boundary register without its paired `TRADE_EDGE` is a contract violation that MUST surface as an audit-log alarm in the per-tick conservation auditor.

**Border Commute Synthesis (FR-031 to FR-036)** *(added 2026-05-13 via Clarifications — Option B scope)*

- **FR-031**: The system MUST provide a `BorderCommuteSynthesisLoader` that produces aggregate Canadian-bound commute flow estimates for the Detroit-Windsor corridor, gated behind the `enable_border_commute_synthesis: bool` configuration flag (default `False` to preserve LODES-only behavior). When the flag is enabled, the loader runs at session initialization alongside the LODES OD matrix loader.
- **FR-032**: The loader MUST consume **BTS Border Crossing Data** monthly CSV files for the Detroit metro ports of entry — Ambassador Bridge and Detroit-Windsor Tunnel — restricted to "Personal Vehicle Passengers" or "Personal Vehicles" measure category for the southbound (entering US) direction. Source URL: `https://data.bts.gov/stories/s/Border-Crossing-Entry-Data/jswi-2e7b/`. Files are placed at `data-trove/border_crossings/bts_border_crossings.csv` as fixture data per Constitution III.4.2.
- **FR-033**: The loader MUST consume **StatCan Frontier Counts** monthly CSV (table `71-607-X2023020` or successor) for the Canada-bound northbound counterpart at the same ports of entry, when present. Source URL: `https://www150.statcan.gc.ca/n1/en/catalogue/71-607-X2023020`. Files are placed at `data-trove/border_crossings/statcan_frontier_counts.csv` as fixture data. If the StatCan file is absent, the loader uses BTS data only and emits a one-time audit log entry noting the asymmetry.
- **FR-034**: The loader MUST apply a `border_commute_share` configuration anchor (default `0.50`, traced to the Workforce WindsorEssex 2017 Cross-Border Employment Report's ~6,120 commuters / ~12K daily personal-vehicle crossings ratio) to derive commuter counts from total vehicle counts. The anchor MUST live in `GameDefines` with a documented citation comment per Constitution III.1 (No Magic Constants).
- **FR-035**: The loader MUST produce one aggregate Canadian-bound flow row per simulated week per direction (Detroit→Windsor and Windsor→Detroit), with magnitude derived as `monthly_vehicle_count × border_commute_share / weeks_per_month`. The flow rows MUST merge into the loaded `LODESYearMatrix` as additional sparse entries with `workplace_dest='canada'`, `workplace_dest_kind='external'` (US→Canada direction) or `workplace_dest=<aggregated tri-county hex set representative>`, such that the existing `Vol2CirculationStep` and `CrossBorderCommuteClassifier` handle them uniformly with native LODES rows. **Week-to-month convention** (Constitution III.7 determinism requirement): week numbering MUST follow ISO 8601 (week 1 contains the first Thursday of the year); the month for week W is the month containing the **Wednesday** of that week (the midpoint day, ensuring deterministic single-month assignment for every week including those crossing month boundaries). `weeks_per_month` is the constant `52 / 12 ≈ 4.333...`.
- **FR-036**: When `enable_border_commute_synthesis = True` AND the BTS source file is missing, initialization MUST fail fast with an explicit error naming the missing file path. When the flag is `False`, missing source files MUST NOT prevent initialization (the synthesis is opt-in).

### Key Entities

- **LODES OD Matrix**: Sparse representation of worker commute flows for a single calendar year, with rows = origin hexes (H3 res-7) and columns = destination hexes or out-of-area destination buckets. Entries are integer worker counts. Loaded at session initialization; immutable for the year. One matrix per simulated year. Constitution II.12 governs the representation.
- **Vol II Circulation Step**: Per-tick engine system that consumes the LODES OD matrix and the current hex `v` vector, produces the next-tick hex `v` vector for in-study-area hexes, and emits `COMMUTE_OUT` boundary register rows for every flow that exits the study area. Sub-stage 5c of the per-tick pipeline.
- **Φ Distribution Engine Wiring**: The integration point where `ImperialRentSystem.step()` invokes `distribute_phi_week_to_counties()` per tick. This is a wiring concern, not a new computational primitive — the helper already exists; the engine system slot does not yet call it.
- **Cross-Border Commute Classification Rule**: The single classifier function that maps a LODES destination identifier to one of `{'canada', 'rest_of_usa', <in-study-area hex H3 index>}` for the purpose of populating `dest_node_id` on emitted boundary register rows.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A one-tick run of the Detroit tri-county scenario produces a non-empty Circulation post-state that differs from the Production post-state in the expected v vector — i.e., variable capital has demonstrably moved between hexes.
- **SC-002**: For every tick of a 52-tick (one simulated year) Detroit run, total in-study-area `v` plus the magnitude of all `COMMUTE_OUT` rows added that tick equals the pre-circulation total within a residual no greater than 1e-9 × pre-circulation total.
- **SC-003**: Over a 52-tick run, the annual sum of `DRAIN_EDGE` magnitudes from each external source equals that source's declared `phi_year_inflow` within a residual no greater than 52 × 1e-9 × phi_year_inflow.
- **SC-004**: For every Windsor-bound LODES OD entry, the emitted boundary register row carries `dest_node_id='canada'`; for every Toledo-bound (or other domestic-out-of-area) entry, the row carries `dest_node_id='rest_of_usa'`. Misclassification rate across a full Detroit LODES year MUST be 0.
- **SC-005**: A re-run of the same Detroit scenario with the same seed produces a bit-identical sequence of Circulation post-states and boundary register rows, demonstrating Constitution III.7 determinism through the new stage.
- **SC-006**: Initialization fails fast (within 5 seconds) with an explicit error message when LODES data is missing for any year the scenario spans, rather than silently proceeding with a no-op Circulation.
- **SC-007**: A 52-tick Detroit run executes the Circulation stage in aggregate wall time no greater than 10% of the existing per-tick budget for the four already-shipped flow stages combined — i.e., adding Circulation does not double the per-tick cost.
- **SC-008**: After 52 ticks of a Detroit session, the boundary flow register contains a non-zero count of rows with `source_kind=external, dest_kind=county, flow_type=DRAIN_EDGE` (proves Φ wiring is live) AND a non-zero count of rows with `source_kind=hex, dest_node_id='canada', flow_type=COMMUTE_OUT` (proves Detroit-Windsor routing is live) AND for every `COMMUTE_OUT` row with `dest_kind=external` there exists a paired `TRADE_EDGE` row with the swapped source/dest and equal magnitude in the same tick (proves wage-repatriation pairing per FR-030a is live).
- **SC-009**: A property-based test exercising 50 random in-study-area hex `v` vectors with a fixed synthetic OD matrix demonstrates conservation (FR-010) holds in 100% of trials within tolerance.
- **SC-010**: The `qa:audit` operator script run against a freshly-initialized Detroit session reports zero `severity='alarm'` audit log entries attributable to the Vol II Circulation stage across 780 ticks (one full Detroit 2010-2025 scenario).
- **SC-011** *(added 2026-05-13 via Clarifications — Option B scope)*: When `enable_border_commute_synthesis = True`, a 52-tick Detroit run produces a non-zero count of `COMMUTE_OUT` boundary register rows with `dest_node_id='canada'` (the synthesis is wired correctly and the classifier routes synthesized rows). The total Canadian-bound `COMMUTE_OUT` magnitude across 52 weeks of one simulated year matches the BTS-derived annual commuter estimate within 5% (allowing for week-to-month aggregation rounding).
- **SC-012** *(added 2026-05-13 via Clarifications — Option B scope)*: When `enable_border_commute_synthesis = False` (default), the engine produces zero `COMMUTE_OUT` rows with `dest_node_id='canada'` from canonical LODES alone (verifying the gap research.md §4 documented), and the FR-026 Canada-must-exist invariant remains active for the synthetic-row test path only.

## Assumptions

- The LODES dataset on disk at `data-trove/lodes/od/` is sufficient for the Detroit tri-county study area for the years 2010–2025. Years missing from the on-disk dataset will be handled by the FR-004 nearest-year clamp; if every year is missing, this becomes a separate data-acquisition task outside this spec's scope.
- The LODES `us_xwalk.csv.gz` crosswalk file at `data-trove/lodes/us_xwalk.csv.gz` is the authoritative block-group-to-hex mapping. If any tri-county block group is missing from the crosswalk, the loader treats the corresponding hex as having no LODES origin/destination presence (consistent with the FR-011 zero-row-sum carry-forward rule).
- The county-level import-share-weighted exposure weights consumed by `distribute_phi_week_to_counties()` are a caller-supplied vector (per spec 062 T058 landing note). Deriving those exposure weights from BEA I-O × QCEW industry mix is a separate concern (spec 062 T057, deferred) — this spec assumes the caller provides them, and treats the weight vector as a configuration input.
- The "Canadian destination" classification at FR-023 hinges on LODES destinations encoding a country code or distinguishable state-code field. If the LODES Detroit-Windsor border subset is published under a non-standard schema (e.g., as a synthetic StatCan augmentation rather than canonical LODES), the classification rule will use whatever discriminator is present; downstream contract details will be locked in `/speckit.plan` Phase 0 research.
- *(Phase 0 finding, 2026-05-13)*: **Canonical LODES does NOT include Canadian destinations** — LODES is a US-only LEHD product. The FR-023 Canadian classification rule is implemented and unit-tested with synthetic Canadian-coded OD rows (state-code prefix outside `{01-56, 60-78}`), but production rows from canonical LODES alone will produce zero `dest_node_id='canada'` boundary register rows. Populating real Detroit-Windsor cross-border commute requires a separate StatCan-or-equivalent ingestion (a downstream spec 064 candidate). The FR-026 fail-fast invariant remains active and is integration-tested by injecting a synthetic Canadian-coded row when the external-node registry omits Canada. See `research.md §4` for the full finding.
- Slime-mold conductivity routing (the emergent informal-economy half of Constitution II.13's Transport Substrate) is out of scope. This spec implements only the deterministic min-cost flow component, leaving slime-mold for a future spec that will treat the LODES OD matrix as a base layer and add conductivity as an overlay (per spec 062 research §6's forward-pointer).
- Per-(hex, NAICS) industry breakdown of variable capital is out of scope. Circulation operates on hex-aggregated `v`; industry breakdown remains derived-on-read per spec 062 FR-031.
- The existing five-flow pipeline ordering (spec 062 FR-025/FR-053) is fixed. This spec inserts Circulation at the already-allocated sub-stage 5c slot; it does not renegotiate the pipeline order.
- The simulation continues to run on PostgreSQL at runtime per spec 037, with SQLite read-only at initialization per spec 062. LODES files are read once at session init (Phase 1 of two-phase persistence) and converted into Postgres-resident `immutable_reference_lodes_*` rows for runtime access — no SQLite reads during a tick.
- Constitution IV.1's "Canada is a first-class boundary node" claim was operationalized by spec 062's external-node bootstrap (T078, landed) — Canada exists in the runtime registry. This spec depends on that bootstrap and refuses to start (FR-026) if the precondition is violated.
